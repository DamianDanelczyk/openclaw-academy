# Memory

Memory is what separates a useful assistant from a forgetful chatbot. OpenClaw uses a **layered memory stack** that mirrors how human memory works: some things are front of mind, some are in the back of your head, and some require a trigger to recall. Understanding these layers lets you get the most out of your assistant's ability to remember — and forget.

## The Memory Stack

OpenClaw's memory is organized into four layers, each with a different purpose and lifespan:

```
┌─────────────────────────────┐
│      Session Memory         │  ← What you just said (minutes)
├─────────────────────────────┤
│      Daily Memory           │  ← Today's highlights (hours)
├─────────────────────────────┤
│      Long-Term Memory       │  ← Important facts (weeks/months)
├─────────────────────────────┤
│      Vector Memory          │  ← Semantic search (forever)
└─────────────────────────────┘
```

Each layer feeds into the next. What matters in the moment lives in session memory. What matters today gets promoted to daily memory. What matters long-term gets persisted. And everything is searchable through vector memory.

### Layer 1: Session Memory

This is your assistant's working memory — the current conversation. Every message, every response, every skill result lives here during an active session.

```yaml
session_memory:
  capacity: 128000 tokens
  strategy: "sliding_window"
  summarize_threshold: 50 messages
  priority_items:
    - user_instructions
    - active_tasks
    - recent_context
  eviction_policy: "oldest_first_with_summary"
```

Session memory is fast and detailed but temporary. When the conversation context gets too large, older messages are summarized and compressed. Your explicit instructions and active tasks are never evicted — they are pinned.

**What it remembers:** Everything in the current conversation.
**How long:** Until the session ends (usually daily reset).
**Example:** "You asked me to draft an email 10 minutes ago. Here's the updated version."

### Layer 2: Daily Memory

At the end of each session, OpenClaw distills the important parts into daily memory. Think of it as your assistant writing in a journal before going to sleep.

```yaml
daily_memory:
  auto_generate: true
  max_entries_per_day: 20
  retention: "30 days"
  format:
    type: "structured"
    fields:
      - summary
      - decisions_made
      - tasks_completed
      - tasks_pending
      - preferences_learned
      - notable_events
```

Here is what a daily memory entry might look like:

```json
{
  "date": "2026-04-21",
  "summary": "Busy work day focused on Q2 launch planning.",
  "decisions_made": [
    "Decided to push the design review to Thursday",
    "Chose Option B for the pricing model"
  ],
  "tasks_completed": [
    "Drafted email to Dr. Chen about timeline",
    "Researched competitor pricing"
  ],
  "tasks_pending": [
    "Follow up with Sarah about mockups",
    "Book restaurant for Jamie's birthday"
  ],
  "preferences_learned": [
    "User prefers tables over bullet lists for comparisons"
  ]
}
```

**What it remembers:** Key events, decisions, and tasks from each day.
**How long:** 30 days by default (configurable).
**Example:** "Yesterday you decided to go with Option B for pricing."

### Layer 3: Long-Term Memory

Some things matter beyond a single day. Long-term memory stores facts, preferences, and significant events that your assistant should know indefinitely.

```yaml
long_term_memory:
  storage: "local_db"
  categories:
    facts:
      description: "Concrete information about the user's world"
      examples: ["Sarah joined the team in January", "Q2 deadline is April 30"]
      retention: "until_invalidated"

    preferences:
      description: "How the user likes things done"
      examples: ["Prefers morning meetings", "Likes detailed code comments"]
      retention: "permanent"

    relationships:
      description: "People and how they connect to the user"
      examples: ["Mike reports to Damian", "Dr. Chen is VP of Product"]
      retention: "until_updated"

    events:
      description: "Significant things that happened"
      examples: ["Got promoted in March 2026", "Launched Project Falcon"]
      retention: "permanent"

  management:
    deduplication: true
    conflict_resolution: "latest_wins"
    max_entries: 10000
```

Long-term memories are structured and tagged. When your agent loads context at the start of a session, it pulls in the most relevant long-term memories based on recent topics.

**What it remembers:** Important facts, preferences, relationships, and milestones.
**How long:** Indefinitely, until explicitly updated or removed.
**Example:** "Your partner Jamie's birthday is March 15. Want me to plan something?"

### Layer 4: Vector Memory

This is the deep archive. Every interaction you have ever had is embedded as a vector (a mathematical representation of meaning) and stored in a searchable database. Your agent does not load all of this into context — that would be impossibly large. Instead, it *searches* this layer when it needs to find something specific.

```yaml
vector_memory:
  engine: "local_embeddings"
  model: "text-embedding-3-small"
  dimensions: 1536
  index:
    type: "hnsw"
    metric: "cosine"
  search:
    default_top_k: 5
    score_threshold: 0.75
    rerank: true
  storage:
    location: "~/.openclaw/vectors/"
    max_size: "5GB"
    compression: true
```

When you ask "What did I say about pricing models last month?" — your agent does not remember this from session or daily memory. It runs a semantic search across your vector memory, finds the relevant conversations, and surfaces them.

**What it remembers:** The semantic content of all past interactions.
**How long:** Forever (until you delete it).
**Example:** "In a conversation on March 12, you discussed three pricing models. Here's what you said about each..."

## Memory in Action

Here is how all four layers work together in a real interaction:

```
You: What should I do about the vendor situation?

OpenClaw's internal process:
1. Session memory: "The user mentioned a vendor problem 20 minutes ago"
2. Daily memory: "Yesterday, user decided to give the vendor one more week"
3. Long-term memory: "This vendor (Acme Corp) has had delivery issues since February"
4. Vector search: Finds 3 past conversations about Acme Corp vendor problems

Response: Based on our conversations, Acme Corp has had recurring delivery
issues since February. Yesterday you gave them one more week. That deadline
would be next Monday. Want me to draft the follow-up email, or should we
start looking at alternative vendors?
```

No single memory layer could have produced that response. It required context from all four.

## Memory Management Commands

You have full control over your agent's memory:

```bash
# See memory stats
openclaw memory stats

# Output:
# Session: 23 messages (18% capacity)
# Daily: 147 entries (last 30 days)
# Long-term: 892 facts stored
# Vector: 2.1GB (12,340 embeddings)

# Search memories
openclaw memory search "vendor pricing"

# Forget something specific
openclaw memory forget --query "salary negotiation details"

# Export all memories
openclaw memory export --format json

# Compact vector memory (removes redundant embeddings)
openclaw memory compact

# Set retention policy
openclaw memory retention --daily 60d --vector unlimited
```

## Memory Strategies

### The Minimalist

Keep memory lean. Good for privacy-focused users.

```yaml
strategy: "minimalist"
session_memory: true
daily_memory:
  enabled: true
  retention: "7 days"
long_term_memory:
  enabled: true
  categories: ["preferences"]  # Only remember preferences
vector_memory:
  enabled: false
```

### The Elephant

Remember everything. Good for power users who want deep recall.

```yaml
strategy: "elephant"
session_memory: true
daily_memory:
  enabled: true
  retention: "365 days"
long_term_memory:
  enabled: true
  categories: ["all"]
  max_entries: 50000
vector_memory:
  enabled: true
  max_size: "20GB"
  retention: "unlimited"
```

### The Professional

Balance between utility and privacy. Remembers work stuff, forgets personal chatter.

```yaml
strategy: "professional"
session_memory: true
daily_memory:
  enabled: true
  retention: "90 days"
  filter:
    include: ["tasks", "decisions", "meetings"]
    exclude: ["casual", "personal"]
long_term_memory:
  enabled: true
  categories: ["facts", "relationships", "preferences"]
vector_memory:
  enabled: true
  max_size: "5GB"
  filter:
    include_topics: ["work", "projects", "professional"]
```

## When to Prune

Memory is not always better with more data. Old, outdated memories can actually confuse your agent. Here are signs it is time to prune:

- Your agent references information that is no longer accurate.
- Search results return too many irrelevant matches.
- Your vector store is growing beyond 5GB without clear benefit.
- Your agent seems "confused" about facts that have changed.

Run `openclaw memory audit` to get a report on stale and potentially outdated memories, then review and clean as needed.

Memory is the backbone of a personalized assistant. Get it right, and your agent feels like it truly knows you. Leave it unmanaged, and it becomes a cluttered attic. The good news? A few minutes of configuration goes a long way.
