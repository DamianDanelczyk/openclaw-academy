# Agents & le Runtime Pi

Quand vous parlez à OpenClaw, vous n'envoyez pas simplement du texte à un modèle IA et vous ne recevez pas du texte en retour. Vous interagissez avec un **agent** — une entité persistante dotée d'une personnalité, d'une mémoire, de compétences et d'objectifs. Le moteur qui fait fonctionner les agents s'appelle le **runtime Pi**.

## Qu'est-ce que le Runtime Pi ?

Le runtime Pi est l'environnement d'exécution où vivent vos agents. Pensez-y comme au système d'exploitation de votre assistant IA. Tout comme le système d'exploitation de votre téléphone gère les applications, les notifications et les tâches en arrière-plan, le runtime Pi gère la réflexion de votre agent, l'accès à la mémoire, l'exécution des compétences et la génération des réponses.

Chaque fois que vous envoyez un message, le runtime Pi :

1. Charge la personnalité de l'agent depuis `SOUL.md`
2. Récupère votre profil depuis `USER.md`
3. Extrait les souvenirs pertinents
4. Vérifie les compétences disponibles
5. Envoie tout — votre message et tout ce contexte — au modèle IA
6. Traite la réponse (en exécutant les appels de compétences, en enregistrant les souvenirs)
7. Vous renvoie la réponse finale

Tout cela se passe en millisecondes. Le runtime Pi est rapide parce qu'il conserve les données fréquemment utilisées en mémoire et ne récupère que ce dont il a besoin.

## Configuration à Agent Unique

La plupart des gens commencent avec un seul agent, et franchement, beaucoup n'en ont jamais besoin de plus. Un agent unique est une seule personnalité IA qui gère tout ce que vous lui demandez.

Voici à quoi ressemble une configuration de base à agent unique :

```yaml
# agents/assistant.yaml
agent:
  name: "Atlas"
  runtime: "pi-v3"
  soul: "./SOUL.md"
  user: "./USER.md"

  model:
    default: "gpt-4o"
    fallback: "gpt-4o-mini"

  skills:
    - calendar
    - email
    - web_search
    - notes

  memory:
    session: true
    long_term: true
    vector_search: true

  behavior:
    proactive: false
    ask_before_acting: true
    verbosity: "balanced"
```

Cela vous donne un agent nommé Atlas qui peut accéder à votre calendrier, à vos e-mails, effectuer des recherches sur le web et prendre des notes. Il attend que vous demandiez avant d'agir (pas d'e-mails surprises) et fournit des réponses de longueur équilibrée.

## Configurations Multi-Agents

C'est là que les choses deviennent intéressantes. Parfois, un seul agent ne suffit pas. Peut-être souhaitez-vous un **agent professionnel** entièrement axé sur le travail et un **agent personnel** décontracté et agréable. Ou peut-être voulez-vous des agents spécialisés qui sont des experts dans des domaines spécifiques.

OpenClaw prend en charge les configurations multi-agents où plusieurs agents peuvent coexister — et même collaborer.

```yaml
# agents/config.yaml
multi_agent:
  enabled: true
  coordinator: "primary"

  agents:
    - name: "Primary"
      role: "coordinator"
      soul: "./souls/primary.md"
      description: "Assistant principal, délègue aux spécialistes"

    - name: "Researcher"
      role: "specialist"
      soul: "./souls/researcher.md"
      skills: [web_search, arxiv, wikipedia]
      description: "Recherche approfondie et vérification des faits"

    - name: "Writer"
      role: "specialist"
      soul: "./souls/writer.md"
      skills: [notes, documents, grammar]
      description: "Rédaction, édition et création de contenu"

    - name: "Coder"
      role: "specialist"
      soul: "./souls/coder.md"
      skills: [code_exec, github, terminal]
      description: "Programmation et tâches techniques"
```

Dans cette configuration, l'agent Primary est celui à qui vous parlez. Lorsque vous posez une question de recherche, Primary la reconnaît et la délègue à l'agent Researcher. Quand vous dites "rédigez-moi un article de blog", Primary délègue à Writer. Vous voyez une seule conversation, mais plusieurs spécialistes travaillent en coulisses.

## Schémas de Coordination des Sous-Agents

Il existe plusieurs façons pour les agents de travailler ensemble. Choisir le bon schéma dépend de ce que vous souhaitez accomplir.

### Le Schéma du Délégateur

Un agent principal délègue des tâches à des spécialistes. C'est le schéma le plus courant.

```yaml
coordination:
  pattern: "delegator"
  primary: "Primary"
  delegation_rules:
    - trigger: "research|find out|look up"
      delegate_to: "Researcher"
      return_to: "Primary"
    - trigger: "write|draft|compose"
      delegate_to: "Writer"
      return_to: "Primary"
```

### Le Schéma Pipeline

Les agents travaillent en séquence, chacun ajoutant à la sortie. Idéal pour les flux de travail de contenu.

```yaml
coordination:
  pattern: "pipeline"
  stages:
    - agent: "Researcher"
      task: "Rassembler les faits et les sources"
      output: "research_brief"
    - agent: "Writer"
      task: "Rédiger le contenu à partir de la recherche"
      input: "research_brief"
      output: "draft"
    - agent: "Primary"
      task: "Réviser et finaliser"
      input: "draft"
      output: "final"
```

Imaginez que vous demandez : "Rédigez-moi un article de blog sur les tendances des énergies renouvelables." Le Researcher rassemble les données actuelles, le Writer les transforme en brouillon, et l'agent Primary peaufine et livre le résultat. Tout cela à partir d'une seule demande.

### Le Schéma de Consensus

Plusieurs agents donnent leur avis, et le coordinateur synthétise leurs perspectives. Utile pour la prise de décision.

```yaml
coordination:
  pattern: "consensus"
  coordinator: "Primary"
  participants:
    - agent: "Optimist"
      perspective: "Se concentrer sur les opportunités et les avantages"
    - agent: "Critic"
      perspective: "Se concentrer sur les risques et les inconvénients"
    - agent: "Pragmatist"
      perspective: "Se concentrer sur la mise en œuvre pratique"
  synthesis:
    method: "coordinator_summary"
    include_dissent: true
```

Demandez "Devrais-je changer de carrière ?" et vous obtenez une réponse réfléchie qui tient compte de multiples angles — pas d'un seul point de vue.

## Configuration du Runtime

Le runtime Pi lui-même dispose de paramètres qui affectent le comportement de tous vos agents :

```yaml
# pi-runtime.yaml
runtime:
  version: "pi-v3"
  concurrency:
    max_parallel_agents: 4
    max_skill_threads: 8
  timeouts:
    agent_response: 30s
    skill_execution: 15s
    total_turn: 60s
  context:
    max_tokens: 128000
    reserve_for_response: 4096
  logging:
    level: "info"
    include_agent_reasoning: false
```

Quelques points importants à noter :

- **max_parallel_agents** — combien d'agents peuvent réfléchir en même temps. Plus de parallélisme signifie des réponses multi-agents plus rapides mais des coûts plus élevés.
- **max_tokens et reserve_for_response** — le runtime gère automatiquement votre fenêtre de contexte. Il réserve de la place pour la réponse afin que le modèle ne manque pas d'espace.
- **include_agent_reasoning** — mettez ceci à `true` si vous voulez voir le processus de réflexion interne de l'agent. Utile pour le débogage, mais bruyant pour un usage quotidien.

## Créer votre Premier Agent

Prêt à configurer un agent ? Voici le chemin le plus rapide :

```bash
# Créer un nouvel agent
openclaw agent create --name "MyAssistant"

# Cela génère :
# agents/myassistant/
#   ├── agent.yaml       (configuration)
#   ├── SOUL.md          (personnalité)
#   └── skills.yaml      (liaisons de compétences)

# Commencer à lui parler
openclaw chat --agent MyAssistant
```

La CLI génère tout automatiquement. Modifiez le `SOUL.md` pour lui donner une personnalité, ajustez `agent.yaml` pour régler le comportement, et vous êtes opérationnel.

## Quand Passer au Multi-Agent

La plupart des gens n'ont pas besoin de configurations multi-agents immédiatement. Voici des signes que vous pourriez en bénéficier :

- Vous vous surprenez à souhaiter que votre assistant ait des personnalités différentes selon les contextes.
- Vos tâches impliquent régulièrement plusieurs étapes distinctes (recherche, puis rédaction, puis révision).
- Vous voulez une expertise spécialisée — un agent qui ne gère que les finances, un autre qui ne gère que la planification.
- Vous construisez un outil d'équipe où différents agents servent différents départements.

Commencez avec un agent. Peaufinez-le. Puis développez quand le mono-agent vous semble limité. Le runtime Pi rend le passage de un à plusieurs agents parfaitement fluide.
