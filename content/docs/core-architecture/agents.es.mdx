# Agentes y el Runtime Pi

Cuando hablas con OpenClaw, no estás simplemente enviando texto a un modelo de IA y recibiendo texto de vuelta. Estás interactuando con un **agente** — una entidad persistente que tiene personalidad, memoria, habilidades y objetivos. El motor que hace funcionar a los agentes se llama **runtime Pi**.

## ¿Qué Es el Runtime Pi?

El runtime Pi es el entorno de ejecución donde viven tus agentes. Piensa en él como el sistema operativo de tu asistente de IA. Así como el sistema operativo de tu teléfono gestiona aplicaciones, notificaciones y tareas en segundo plano, el runtime Pi gestiona el pensamiento de tu agente, el acceso a la memoria, la ejecución de habilidades y la generación de respuestas.

Cada vez que envías un mensaje, el runtime Pi:

1. Carga la personalidad del agente desde `SOUL.md`
2. Incorpora tu perfil desde `USER.md`
3. Recupera memorias relevantes
4. Verifica las habilidades disponibles
5. Envía todo — tu mensaje más todo ese contexto — al modelo de IA
6. Procesa la respuesta (ejecutando llamadas de habilidades, almacenando memorias)
7. Te devuelve la respuesta final

Todo esto ocurre en milisegundos. El runtime Pi es rápido porque mantiene los datos de uso frecuente en memoria y solo obtiene lo que necesita.

## Configuración de Agente Único

La mayoría de las personas comienzan con un solo agente y, honestamente, muchos nunca necesitan más. Un agente único es una personalidad de IA que maneja todo lo que le planteas.

Así es como se ve una configuración básica de agente único:

```yaml
# agents/assistant.yaml
agent:
  name: "Atlas"
  runtime: "pi-v3"
  soul: "./SOUL.md"
  user: "./USER.md"

  model:
    default: "gpt-4o"
    fallback: "gpt-4o-mini"

  skills:
    - calendar
    - email
    - web_search
    - notes

  memory:
    session: true
    long_term: true
    vector_search: true

  behavior:
    proactive: false
    ask_before_acting: true
    verbosity: "balanced"
```

Esto te da un agente llamado Atlas que puede acceder a tu calendario, correo electrónico, buscar en la web y tomar notas. Espera a que le pidas antes de hacer cosas (sin correos electrónicos sorpresa) y da respuestas de longitud equilibrada.

## Configuraciones Multiagente

Aquí es donde las cosas se ponen interesantes. A veces un solo agente no es suficiente. Quizás quieras un **agente de trabajo** que sea todo negocios y un **agente personal** que sea casual y divertido. O quizás quieras agentes especializados que sean expertos en áreas específicas.

OpenClaw soporta configuraciones multiagente donde múltiples agentes pueden coexistir — e incluso colaborar.

```yaml
# agents/config.yaml
multi_agent:
  enabled: true
  coordinator: "primary"

  agents:
    - name: "Primary"
      role: "coordinator"
      soul: "./souls/primary.md"
      description: "Asistente principal, delega a especialistas"

    - name: "Researcher"
      role: "specialist"
      soul: "./souls/researcher.md"
      skills: [web_search, arxiv, wikipedia]
      description: "Investigación profunda y verificación de hechos"

    - name: "Writer"
      role: "specialist"
      soul: "./souls/writer.md"
      skills: [notes, documents, grammar]
      description: "Escritura, edición y creación de contenido"

    - name: "Coder"
      role: "specialist"
      soul: "./souls/coder.md"
      skills: [code_exec, github, terminal]
      description: "Programación y tareas técnicas"
```

En esta configuración, el agente Primary es con el que hablas. Cuando haces una pregunta de investigación, Primary la reconoce y se la pasa al agente Researcher. Cuando dices "escríbeme una publicación de blog", Primary delega al Writer. Ves una sola conversación, pero múltiples especialistas están trabajando entre bastidores.

## Patrones de Coordinación de Sub-Agentes

Hay varias formas en que los agentes pueden trabajar juntos. Elegir el patrón correcto depende de lo que intentas lograr.

### El Patrón Delegador

Un agente jefe delega tareas a especialistas. Este es el patrón más común.

```yaml
coordination:
  pattern: "delegator"
  primary: "Primary"
  delegation_rules:
    - trigger: "investigar|encontrar|buscar"
      delegate_to: "Researcher"
      return_to: "Primary"
    - trigger: "escribir|redactar|componer"
      delegate_to: "Writer"
      return_to: "Primary"
```

### El Patrón de Tubería

Los agentes trabajan en secuencia, cada uno añadiendo a la salida. Excelente para flujos de trabajo de contenido.

```yaml
coordination:
  pattern: "pipeline"
  stages:
    - agent: "Researcher"
      task: "Recopilar hechos y fuentes"
      output: "research_brief"
    - agent: "Writer"
      task: "Redactar contenido a partir de la investigación"
      input: "research_brief"
      output: "draft"
    - agent: "Primary"
      task: "Revisar y finalizar"
      input: "draft"
      output: "final"
```

Imagina preguntar: "Escríbeme una publicación de blog sobre tendencias de energía renovable." El Researcher recopila datos actuales, el Writer los convierte en un borrador y el agente Primary lo pule y entrega. Todo a partir de una solicitud.

### El Patrón de Consenso

Múltiples agentes opinan y el coordinador sintetiza sus perspectivas. Útil para la toma de decisiones.

```yaml
coordination:
  pattern: "consensus"
  coordinator: "Primary"
  participants:
    - agent: "Optimist"
      perspective: "Centrarse en oportunidades y beneficios"
    - agent: "Critic"
      perspective: "Centrarse en riesgos e inconvenientes"
    - agent: "Pragmatist"
      perspective: "Centrarse en la implementación práctica"
  synthesis:
    method: "coordinator_summary"
    include_dissent: true
```

Pregunta "¿Debería cambiar de carrera?" y obtienes una respuesta reflexiva que considera múltiples ángulos — no solo un punto de vista.

## Configuración del Runtime

El runtime Pi en sí tiene configuraciones que afectan cómo se comportan todos tus agentes:

```yaml
# pi-runtime.yaml
runtime:
  version: "pi-v3"
  concurrency:
    max_parallel_agents: 4
    max_skill_threads: 8
  timeouts:
    agent_response: 30s
    skill_execution: 15s
    total_turn: 60s
  context:
    max_tokens: 128000
    reserve_for_response: 4096
  logging:
    level: "info"
    include_agent_reasoning: false
```

Algunas cosas que vale la pena saber:

- **max_parallel_agents** — cuántos agentes pueden pensar al mismo tiempo. Más paralelismo significa respuestas multiagente más rápidas pero mayor costo.
- **max_tokens y reserve_for_response** — el runtime gestiona tu ventana de contexto automáticamente. Reserva espacio para la respuesta para que el modelo no se quede sin margen.
- **include_agent_reasoning** — establécelo en `true` si quieres ver el proceso de pensamiento interno del agente. Útil para depuración, ruidoso para el uso cotidiano.

## Creando Tu Primer Agente

¿Listo para configurar un agente? Aquí está el camino más rápido:

```bash
# Crear un nuevo agente
openclaw agent create --name "MyAssistant"

# Esto genera:
# agents/myassistant/
#   ├── agent.yaml       (configuración)
#   ├── SOUL.md          (personalidad)
#   └── skills.yaml      (vinculaciones de habilidades)

# Empezar a hablar con él
openclaw chat --agent MyAssistant
```

La CLI genera todo por ti. Edita el `SOUL.md` para darle personalidad, ajusta `agent.yaml` para modificar el comportamiento y ya estás en marcha.

## Cuándo Usar Múltiples Agentes

La mayoría de las personas no necesitan configuraciones multiagente de inmediato. Aquí hay señales de que podrías beneficiarte de una:

- Deseas que tu asistente tenga diferentes personalidades para diferentes contextos.
- Tus tareas regularmente implican múltiples pasos distintos (investigar, luego escribir, luego revisar).
- Quieres experiencia especializada — un agente que solo maneja finanzas, otro que solo maneja programación.
- Estás construyendo una herramienta de equipo donde diferentes agentes sirven a diferentes departamentos.

Comienza con un agente. Ajústalo a la perfección. Luego expande cuando el agente único se sienta limitante. El runtime Pi hace que escalar de uno a muchos sea fluido.
