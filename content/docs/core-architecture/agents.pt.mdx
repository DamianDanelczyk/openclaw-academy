# Agentes & o Runtime Pi

Quando você fala com o OpenClaw, não está apenas enviando texto para um modelo de IA e recebendo texto de volta. Você está interagindo com um **agente** — uma entidade persistente que tem personalidade, memória, habilidades e objetivos. O motor que faz os agentes funcionarem é chamado de **runtime Pi**.

## O Que É o Runtime Pi?

O runtime Pi é o ambiente de execução onde seus agentes vivem. Pense nele como o sistema operacional do seu assistente de IA. Assim como o SO do seu celular gerencia apps, notificações e tarefas em segundo plano, o runtime Pi gerencia o raciocínio do seu agente, o acesso à memória, a execução de habilidades e a geração de respostas.

Toda vez que você envia uma mensagem, o runtime Pi:

1. Carrega a personalidade do agente a partir do `SOUL.md`
2. Puxa o seu perfil do `USER.md`
3. Recupera memórias relevantes
4. Verifica as habilidades disponíveis
5. Envia tudo — sua mensagem mais todo esse contexto — para o modelo de IA
6. Processa a resposta (executando chamadas de habilidades, armazenando memórias)
7. Retorna a resposta final para você

Tudo isso acontece em milissegundos. O runtime Pi é rápido porque mantém dados frequentemente usados na memória e só busca o que precisa.

## Configuração de Agente Único

A maioria das pessoas começa com um único agente e, honestamente, muitos nunca precisam de mais. Um único agente é uma personalidade de IA que lida com tudo que você jogar nela.

Aqui está como uma configuração básica de agente único se parece:

```yaml
# agents/assistant.yaml
agent:
  name: "Atlas"
  runtime: "pi-v3"
  soul: "./SOUL.md"
  user: "./USER.md"

  model:
    default: "gpt-4o"
    fallback: "gpt-4o-mini"

  skills:
    - calendar
    - email
    - web_search
    - notes

  memory:
    session: true
    long_term: true
    vector_search: true

  behavior:
    proactive: false
    ask_before_acting: true
    verbosity: "balanced"
```

Isso lhe dá um agente chamado Atlas que pode acessar seu calendário, e-mail, pesquisar na web e fazer anotações. Ele espera você perguntar antes de fazer coisas (sem e-mails surpresa) e dá respostas de tamanho equilibrado.

## Configurações de Múltiplos Agentes

É aqui que as coisas ficam interessantes. Às vezes um agente não é suficiente. Talvez você queira um **agente de trabalho** que seja todo negócios e um **agente pessoal** que seja casual e divertido. Ou talvez você queira agentes especializados que sejam especialistas em áreas específicas.

O OpenClaw suporta configurações de múltiplos agentes onde vários agentes podem coexistir — e até colaborar.

```yaml
# agents/config.yaml
multi_agent:
  enabled: true
  coordinator: "primary"

  agents:
    - name: "Primary"
      role: "coordinator"
      soul: "./souls/primary.md"
      description: "Assistente principal, delega para especialistas"

    - name: "Researcher"
      role: "specialist"
      soul: "./souls/researcher.md"
      skills: [web_search, arxiv, wikipedia]
      description: "Pesquisa aprofundada e verificação de fatos"

    - name: "Writer"
      role: "specialist"
      soul: "./souls/writer.md"
      skills: [notes, documents, grammar]
      description: "Escrita, edição e criação de conteúdo"

    - name: "Coder"
      role: "specialist"
      soul: "./souls/coder.md"
      skills: [code_exec, github, terminal]
      description: "Programação e tarefas técnicas"
```

Nessa configuração, o agente Primary é com quem você fala. Quando você faz uma pergunta de pesquisa, o Primary reconhece e passa para o agente Researcher. Quando você diz "escreva um post para o blog", o Primary delega para o Writer. Você vê uma conversa, mas vários especialistas estão trabalhando nos bastidores.

## Padrões de Coordenação de Sub-Agentes

Existem várias maneiras de os agentes trabalharem juntos. Escolher o padrão certo depende do que você está tentando realizar.

### O Padrão Delegador

Um agente chefe delega tarefas a especialistas. Esse é o padrão mais comum.

```yaml
coordination:
  pattern: "delegator"
  primary: "Primary"
  delegation_rules:
    - trigger: "pesquisa|descubra|procure"
      delegate_to: "Researcher"
      return_to: "Primary"
    - trigger: "escreva|rascunho|componha"
      delegate_to: "Writer"
      return_to: "Primary"
```

### O Padrão Pipeline

Agentes trabalham em sequência, cada um adicionando ao resultado. Ótimo para fluxos de trabalho de conteúdo.

```yaml
coordination:
  pattern: "pipeline"
  stages:
    - agent: "Researcher"
      task: "Reunir fatos e fontes"
      output: "research_brief"
    - agent: "Writer"
      task: "Rascunhar conteúdo a partir da pesquisa"
      input: "research_brief"
      output: "draft"
    - agent: "Primary"
      task: "Revisar e finalizar"
      input: "draft"
      output: "final"
```

Imagine pedir: "Escreva um post sobre tendências de energia renovável." O Researcher reúne dados atuais, o Writer transforma em rascunho e o agente Primary aprimora e entrega. Tudo a partir de uma solicitação.

### O Padrão Consenso

Vários agentes dão suas opiniões e o coordenador sintetiza as perspectivas. Útil para tomada de decisões.

```yaml
coordination:
  pattern: "consensus"
  coordinator: "Primary"
  participants:
    - agent: "Optimist"
      perspective: "Focar em oportunidades e benefícios"
    - agent: "Critic"
      perspective: "Focar em riscos e desvantagens"
    - agent: "Pragmatist"
      perspective: "Focar na implementação prática"
  synthesis:
    method: "coordinator_summary"
    include_dissent: true
```

Pergunte "Devo mudar de carreira?" e receba uma resposta ponderada que considera múltiplos ângulos — não apenas um ponto de vista.

## Configuração do Runtime

O próprio runtime Pi tem configurações que afetam como todos os seus agentes se comportam:

```yaml
# pi-runtime.yaml
runtime:
  version: "pi-v3"
  concurrency:
    max_parallel_agents: 4
    max_skill_threads: 8
  timeouts:
    agent_response: 30s
    skill_execution: 15s
    total_turn: 60s
  context:
    max_tokens: 128000
    reserve_for_response: 4096
  logging:
    level: "info"
    include_agent_reasoning: false
```

Algumas coisas que vale saber:

- **max_parallel_agents** — quantos agentes podem pensar ao mesmo tempo. Mais paralelismo significa respostas multi-agente mais rápidas, mas custo maior.
- **max_tokens e reserve_for_response** — o runtime gerencia sua janela de contexto automaticamente. Ele reserva espaço para a resposta para que o modelo não fique sem espaço.
- **include_agent_reasoning** — defina como `true` se quiser ver o processo de raciocínio interno do agente. Útil para depuração, barulhento para uso diário.

## Criando Seu Primeiro Agente

Pronto para configurar um agente? Aqui está o caminho mais rápido:

```bash
# Criar um novo agente
openclaw agent create --name "MyAssistant"

# Isso gera:
# agents/myassistant/
#   ├── agent.yaml       (configuração)
#   ├── SOUL.md          (personalidade)
#   └── skills.yaml      (vínculos de habilidades)

# Começar a conversar com ele
openclaw chat --agent MyAssistant
```

A CLI cria tudo para você. Edite o `SOUL.md` para dar personalidade, ajuste o `agent.yaml` para modificar o comportamento e você está pronto para usar.

## Quando Usar Múltiplos Agentes

A maioria das pessoas não precisa de configurações multi-agente imediatamente. Aqui estão sinais de que você pode se beneficiar de uma:

- Você deseja que seu assistente tivesse personalidades diferentes para contextos diferentes.
- Suas tarefas regularmente envolvem múltiplas etapas distintas (pesquisa, depois escrita, depois revisão).
- Você quer expertise especializada — um agente que só lida com finanças, outro que só lida com agendamento.
- Você está construindo uma ferramenta de equipe onde diferentes agentes atendem diferentes departamentos.

Comece com um agente. Ajuste-o bem. Depois expanda quando um único agente parecer limitante. O runtime Pi torna a escalada de um para muitos perfeita.
