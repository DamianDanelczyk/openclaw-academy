# Agents & the Pi Runtime

When you talk to OpenClaw, you are not just sending text to an AI model and getting text back. You are interacting with an **agent** — a persistent entity that has personality, memory, skills, and goals. The engine that makes agents tick is called the **Pi runtime**.

## What Is the Pi Runtime?

The Pi runtime is the execution environment where your agents live. Think of it like the operating system for your AI assistant. Just as your phone's OS manages apps, notifications, and background tasks, the Pi runtime manages your agent's thinking, memory access, skill execution, and response generation.

Every time you send a message, the Pi runtime:

1. Loads the agent's personality from `SOUL.md`
2. Pulls in your profile from `USER.md`
3. Retrieves relevant memories
4. Checks available skills
5. Sends everything — your message plus all that context — to the AI model
6. Processes the response (running any skill calls, storing memories)
7. Returns the final answer to you

All of this happens in milliseconds. The Pi runtime is fast because it keeps frequently-used data in memory and only fetches what it needs.

## Single Agent Setup

Most people start with a single agent, and honestly, many never need more. A single agent is one AI personality that handles everything you throw at it.

Here is what a basic single-agent configuration looks like:

```yaml
# agents/assistant.yaml
agent:
  name: "Atlas"
  runtime: "pi-v3"
  soul: "./SOUL.md"
  user: "./USER.md"

  model:
    default: "gpt-4o"
    fallback: "gpt-4o-mini"

  skills:
    - calendar
    - email
    - web_search
    - notes

  memory:
    session: true
    long_term: true
    vector_search: true

  behavior:
    proactive: false
    ask_before_acting: true
    verbosity: "balanced"
```

This gives you an agent named Atlas that can access your calendar, email, search the web, and take notes. It waits for you to ask before doing things (no surprise emails), and gives balanced-length responses.

## Multi-Agent Setups

Here is where things get interesting. Sometimes one agent is not enough. Maybe you want a **work agent** that is all business and a **personal agent** that is casual and fun. Or maybe you want specialized agents that are experts in specific areas.

OpenClaw supports multi-agent setups where multiple agents can coexist — and even collaborate.

```yaml
# agents/config.yaml
multi_agent:
  enabled: true
  coordinator: "primary"

  agents:
    - name: "Primary"
      role: "coordinator"
      soul: "./souls/primary.md"
      description: "Main assistant, delegates to specialists"

    - name: "Researcher"
      role: "specialist"
      soul: "./souls/researcher.md"
      skills: [web_search, arxiv, wikipedia]
      description: "Deep research and fact-checking"

    - name: "Writer"
      role: "specialist"
      soul: "./souls/writer.md"
      skills: [notes, documents, grammar]
      description: "Writing, editing, and content creation"

    - name: "Coder"
      role: "specialist"
      soul: "./souls/coder.md"
      skills: [code_exec, github, terminal]
      description: "Programming and technical tasks"
```

In this setup, the Primary agent is the one you talk to. When you ask a research question, Primary recognizes it and hands it off to the Researcher agent. When you say "write me a blog post," Primary delegates to the Writer. You see one conversation, but multiple specialists are working behind the scenes.

## Sub-Agent Coordination Patterns

There are several ways agents can work together. Picking the right pattern depends on what you are trying to accomplish.

### The Delegator Pattern

One boss agent delegates tasks to specialists. This is the most common pattern.

```yaml
coordination:
  pattern: "delegator"
  primary: "Primary"
  delegation_rules:
    - trigger: "research|find out|look up"
      delegate_to: "Researcher"
      return_to: "Primary"
    - trigger: "write|draft|compose"
      delegate_to: "Writer"
      return_to: "Primary"
```

### The Pipeline Pattern

Agents work in sequence, each adding to the output. Great for content workflows.

```yaml
coordination:
  pattern: "pipeline"
  stages:
    - agent: "Researcher"
      task: "Gather facts and sources"
      output: "research_brief"
    - agent: "Writer"
      task: "Draft content from research"
      input: "research_brief"
      output: "draft"
    - agent: "Primary"
      task: "Review and finalize"
      input: "draft"
      output: "final"
```

Imagine asking: "Write me a blog post about renewable energy trends." The Researcher gathers current data, the Writer turns it into a draft, and the Primary agent polishes and delivers it. All from one request.

### The Consensus Pattern

Multiple agents weigh in, and the coordinator synthesizes their perspectives. Useful for decision-making.

```yaml
coordination:
  pattern: "consensus"
  coordinator: "Primary"
  participants:
    - agent: "Optimist"
      perspective: "Focus on opportunities and benefits"
    - agent: "Critic"
      perspective: "Focus on risks and downsides"
    - agent: "Pragmatist"
      perspective: "Focus on practical implementation"
  synthesis:
    method: "coordinator_summary"
    include_dissent: true
```

Ask "Should I switch careers?" and you get a thoughtful answer that considers multiple angles — not just one viewpoint.

## Runtime Configuration

The Pi runtime itself has settings that affect how all your agents behave:

```yaml
# pi-runtime.yaml
runtime:
  version: "pi-v3"
  concurrency:
    max_parallel_agents: 4
    max_skill_threads: 8
  timeouts:
    agent_response: 30s
    skill_execution: 15s
    total_turn: 60s
  context:
    max_tokens: 128000
    reserve_for_response: 4096
  logging:
    level: "info"
    include_agent_reasoning: false
```

A few things worth knowing:

- **max_parallel_agents** — how many agents can think at the same time. More parallelism means faster multi-agent responses but higher cost.
- **max_tokens and reserve_for_response** — the runtime manages your context window automatically. It reserves space for the response so the model does not run out of room.
- **include_agent_reasoning** — set this to `true` if you want to see the agent's internal thought process. Helpful for debugging, noisy for everyday use.

## Creating Your First Agent

Ready to set up an agent? Here is the quickest path:

```bash
# Create a new agent
openclaw agent create --name "MyAssistant"

# This generates:
# agents/myassistant/
#   ├── agent.yaml       (configuration)
#   ├── SOUL.md          (personality)
#   └── skills.yaml      (skill bindings)

# Start talking to it
openclaw chat --agent MyAssistant
```

The CLI scaffolds everything for you. Edit the `SOUL.md` to give it personality, tweak `agent.yaml` to adjust behavior, and you are running.

## When to Go Multi-Agent

Most people do not need multi-agent setups right away. Here are signs you might benefit from one:

- You find yourself wishing your assistant had different personalities for different contexts.
- Your tasks regularly involve multiple distinct steps (research, then write, then review).
- You want specialized expertise — an agent that only handles finances, another that only handles scheduling.
- You are building a team tool where different agents serve different departments.

Start with one agent. Get it dialed in. Then expand when single-agent feels limiting. The Pi runtime makes scaling from one to many seamless.
