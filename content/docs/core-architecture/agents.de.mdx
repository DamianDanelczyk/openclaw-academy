# Agenten & die Pi-Runtime

Wenn du mit OpenClaw sprichst, sendest du nicht einfach Text an ein KI-Modell und bekommst Text zurück. Du interagierst mit einem **Agenten** — einer persistenten Einheit, die Persönlichkeit, Gedächtnis, Fähigkeiten und Ziele hat. Die Engine, die Agenten antreibt, heißt **Pi-Runtime**.

## Was ist die Pi-Runtime?

Die Pi-Runtime ist die Ausführungsumgebung, in der deine Agenten leben. Stell sie dir wie das Betriebssystem für deinen KI-Assistenten vor. So wie das Betriebssystem deines Telefons Apps, Benachrichtigungen und Hintergrundaufgaben verwaltet, verwaltet die Pi-Runtime das Denken, den Speicherzugriff, die Skill-Ausführung und die Antwortgenerierung deines Agenten.

Jedes Mal, wenn du eine Nachricht sendest, erledigt die Pi-Runtime Folgendes:

1. Lädt die Persönlichkeit des Agenten aus `SOUL.md`
2. Holt dein Profil aus `USER.md`
3. Ruft relevante Erinnerungen ab
4. Prüft verfügbare Skills
5. Sendet alles — deine Nachricht plus den gesamten Kontext — an das KI-Modell
6. Verarbeitet die Antwort (führt Skill-Aufrufe aus, speichert Erinnerungen)
7. Gibt die endgültige Antwort an dich zurück

All das geschieht in Millisekunden. Die Pi-Runtime ist schnell, weil sie häufig verwendete Daten im Speicher hält und nur das abruft, was benötigt wird.

## Einzel-Agenten-Setup

Die meisten Menschen beginnen mit einem einzigen Agenten, und ehrlich gesagt brauchen viele nie mehr. Ein einzelner Agent ist eine KI-Persönlichkeit, die alles erledigt, was du ihr zuwirfst.

So sieht eine einfache Einzel-Agenten-Konfiguration aus:

```yaml
# agents/assistant.yaml
agent:
  name: "Atlas"
  runtime: "pi-v3"
  soul: "./SOUL.md"
  user: "./USER.md"

  model:
    default: "gpt-4o"
    fallback: "gpt-4o-mini"

  skills:
    - calendar
    - email
    - web_search
    - notes

  memory:
    session: true
    long_term: true
    vector_search: true

  behavior:
    proactive: false
    ask_before_acting: true
    verbosity: "balanced"
```

Das gibt dir einen Agenten namens Atlas, der auf deinen Kalender, deine E-Mails zugreifen, im Web suchen und Notizen machen kann. Er wartet, bis du ihn fragst, bevor er etwas unternimmt (keine unerwarteten E-Mails), und gibt ausgewogen lange Antworten.

## Multi-Agenten-Setups

Hier wird es interessant. Manchmal reicht ein Agent nicht aus. Vielleicht möchtest du einen **Arbeits-Agenten**, der ganz auf Geschäftliches ausgerichtet ist, und einen **persönlichen Agenten**, der locker und lustig ist. Oder du möchtest spezialisierte Agenten, die Experten in bestimmten Bereichen sind.

OpenClaw unterstützt Multi-Agenten-Setups, bei denen mehrere Agenten nebeneinander existieren — und sogar zusammenarbeiten können.

```yaml
# agents/config.yaml
multi_agent:
  enabled: true
  coordinator: "primary"

  agents:
    - name: "Primary"
      role: "coordinator"
      soul: "./souls/primary.md"
      description: "Hauptassistent, delegiert an Spezialisten"

    - name: "Researcher"
      role: "specialist"
      soul: "./souls/researcher.md"
      skills: [web_search, arxiv, wikipedia]
      description: "Tiefgreifende Recherche und Faktenprüfung"

    - name: "Writer"
      role: "specialist"
      soul: "./souls/writer.md"
      skills: [notes, documents, grammar]
      description: "Schreiben, Bearbeiten und Content-Erstellung"

    - name: "Coder"
      role: "specialist"
      soul: "./souls/coder.md"
      skills: [code_exec, github, terminal]
      description: "Programmierung und technische Aufgaben"
```

In diesem Setup ist der Primary-Agent derjenige, mit dem du sprichst. Wenn du eine Recherchefrage stellst, erkennt Primary dies und leitet sie an den Researcher-Agenten weiter. Wenn du sagst "Schreib mir einen Blogbeitrag", delegiert Primary an den Writer. Du siehst ein einziges Gespräch, aber mehrere Spezialisten arbeiten im Hintergrund.

## Koordinationsmuster für Sub-Agenten

Es gibt mehrere Möglichkeiten, wie Agenten zusammenarbeiten können. Das richtige Muster zu wählen hängt davon ab, was du erreichen möchtest.

### Das Delegator-Muster

Ein Chef-Agent delegiert Aufgaben an Spezialisten. Das ist das häufigste Muster.

```yaml
coordination:
  pattern: "delegator"
  primary: "Primary"
  delegation_rules:
    - trigger: "research|find out|look up"
      delegate_to: "Researcher"
      return_to: "Primary"
    - trigger: "write|draft|compose"
      delegate_to: "Writer"
      return_to: "Primary"
```

### Das Pipeline-Muster

Agenten arbeiten nacheinander, jeder ergänzt die Ausgabe. Großartig für Content-Workflows.

```yaml
coordination:
  pattern: "pipeline"
  stages:
    - agent: "Researcher"
      task: "Fakten und Quellen zusammentragen"
      output: "research_brief"
    - agent: "Writer"
      task: "Inhalte aus der Recherche erstellen"
      input: "research_brief"
      output: "draft"
    - agent: "Primary"
      task: "Überprüfen und finalisieren"
      input: "draft"
      output: "final"
```

Stell dir vor, du fragst: "Schreib mir einen Blogbeitrag über Trends bei erneuerbaren Energien." Der Researcher sammelt aktuelle Daten, der Writer erstellt daraus einen Entwurf, und der Primary-Agent poliert und liefert ihn. Alles aus einer einzigen Anfrage.

### Das Konsens-Muster

Mehrere Agenten bringen ihre Perspektiven ein, und der Koordinator fasst sie zusammen. Nützlich für Entscheidungsfindungen.

```yaml
coordination:
  pattern: "consensus"
  coordinator: "Primary"
  participants:
    - agent: "Optimist"
      perspective: "Konzentriere dich auf Chancen und Vorteile"
    - agent: "Critic"
      perspective: "Konzentriere dich auf Risiken und Nachteile"
    - agent: "Pragmatist"
      perspective: "Konzentriere dich auf praktische Umsetzung"
  synthesis:
    method: "coordinator_summary"
    include_dissent: true
```

Frage "Sollte ich den Beruf wechseln?" und du erhältst eine durchdachte Antwort, die mehrere Perspektiven berücksichtigt — nicht nur einen einzigen Standpunkt.

## Runtime-Konfiguration

Die Pi-Runtime selbst hat Einstellungen, die beeinflussen, wie sich all deine Agenten verhalten:

```yaml
# pi-runtime.yaml
runtime:
  version: "pi-v3"
  concurrency:
    max_parallel_agents: 4
    max_skill_threads: 8
  timeouts:
    agent_response: 30s
    skill_execution: 15s
    total_turn: 60s
  context:
    max_tokens: 128000
    reserve_for_response: 4096
  logging:
    level: "info"
    include_agent_reasoning: false
```

Einige wichtige Hinweise:

- **max_parallel_agents** — wie viele Agenten gleichzeitig denken können. Mehr Parallelismus bedeutet schnellere Multi-Agenten-Antworten, aber höhere Kosten.
- **max_tokens und reserve_for_response** — die Runtime verwaltet dein Kontextfenster automatisch. Sie reserviert Platz für die Antwort, damit dem Modell kein Platz ausgeht.
- **include_agent_reasoning** — setze dies auf `true`, wenn du den internen Denkprozess des Agenten sehen möchtest. Hilfreich für das Debugging, aber lästig im Alltagsgebrauch.

## Deinen ersten Agenten erstellen

Bereit, einen Agenten einzurichten? Hier ist der schnellste Weg:

```bash
# Einen neuen Agenten erstellen
openclaw agent create --name "MyAssistant"

# Das erstellt:
# agents/myassistant/
#   ├── agent.yaml       (Konfiguration)
#   ├── SOUL.md          (Persönlichkeit)
#   └── skills.yaml      (Skill-Bindungen)

# Gespräch starten
openclaw chat --agent MyAssistant
```

Die CLI erstellt alles für dich. Bearbeite die `SOUL.md`, um dem Agenten Persönlichkeit zu geben, passe `agent.yaml` an, um das Verhalten einzustellen, und du bist startklar.

## Wann ein Multi-Agenten-Setup sinnvoll ist

Die meisten Menschen brauchen nicht sofort Multi-Agenten-Setups. Hier sind Anzeichen, dass du davon profitieren könntest:

- Du wünschst dir, dein Assistent hätte unterschiedliche Persönlichkeiten für unterschiedliche Kontexte.
- Deine Aufgaben beinhalten regelmäßig mehrere unterschiedliche Schritte (recherchieren, dann schreiben, dann überprüfen).
- Du möchtest spezialisiertes Fachwissen — ein Agent, der sich nur um Finanzen kümmert, ein anderer nur um Terminplanung.
- Du baust ein Team-Tool, bei dem verschiedene Agenten verschiedenen Abteilungen dienen.

Fange mit einem Agenten an. Stimme ihn genau ab. Erweitere dann, wenn ein einzelner Agent einschränkend wird. Die Pi-Runtime macht das Skalieren von einem zu vielen nahtlos.
