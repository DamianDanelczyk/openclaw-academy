# Sandboxing

Quando seu assistente OpenClaw executa uma ação — lendo um arquivo, chamando uma API, executando um script — você provavelmente quer garantir que não possa acidentalmente (ou maliciosamente) alcançar além do que deveria. É exatamente isso que o sandboxing faz. Ele traça um limite ao redor do ambiente de execução do seu assistente para que as ações permaneçam contidas e previsíveis.

Pense nisso como um cercadinho para o seu assistente. Dentro do sandbox, ele tem tudo que precisa para fazer seu trabalho. Fora do sandbox, seu sistema, arquivos e rede estão fora dos limites.

## Como o Sandboxing Funciona

Por baixo dos panos, o OpenClaw usa isolamento no nível do sistema operacional para criar um ambiente restrito para cada execução de ação. Aqui está o que acontece quando seu assistente executa uma ferramenta:

1. O OpenClaw cria um container temporário e isolado.
2. A ação é executada dentro desse container com permissões limitadas.
3. O container tem acesso apenas a recursos explicitamente permitidos (arquivos, endpoints de rede, variáveis de ambiente).
4. Quando a ação termina, o container é destruído junto com quaisquer arquivos temporários que criou.

Isso significa que mesmo que uma ação tente fazer algo que não deveria — como ler arquivos fora do seu diretório de trabalho ou se conectar a um servidor inesperado — o sandbox a bloqueia.

## Modos de Sandbox

O OpenClaw oferece três modos de sandbox, do mais restritivo ao menos:

### Modo Estrito (Padrão Recomendado)

```yaml
security:
  sandbox:
    enabled: true
    mode: "strict"
```

No modo estrito:
- Sem acesso ao sistema de arquivos fora do diretório de trabalho designado
- Sem acesso à rede, a menos que explicitamente permitido em sua configuração de rede
- Sem acesso a variáveis de ambiente, a menos que explicitamente listadas
- Sem capacidade de gerar processos filhos
- Memória e tempo de CPU são limitados

Esta é a escolha certa para a maioria dos usuários. Comece aqui e só relaxe restrições se tiver um motivo específico.

### Modo Padrão

```yaml
security:
  sandbox:
    enabled: true
    mode: "standard"
```

No modo padrão:
- Acesso somente leitura ao sistema de arquivos (mas não a diretórios sensíveis como `/etc`, diretórios home ou chaves SSH)
- O acesso à rede segue sua configuração de lista de permissões
- Variáveis de ambiente listadas em `sandbox.allowed_env` são acessíveis
- Processos filhos são permitidos mas monitorados
- Limites de memória e CPU ainda se aplicam

O modo padrão é apropriado quando seu assistente precisa ler arquivos de projeto ou dados de configuração como parte de suas tarefas.

### Modo Permissivo

```yaml
security:
  sandbox:
    enabled: true
    mode: "permissive"
```

No modo permissivo:
- Acesso de leitura e escrita a diretórios permitidos
- O acesso à rede segue sua lista de permissões
- Todas as variáveis de ambiente são acessíveis
- Processos filhos são permitidos
- Limites mais altos de memória e CPU

Use o modo permissivo apenas quando seu assistente precisar escrever arquivos, instalar pacotes ou executar tarefas de desenvolvimento. Mesmo no modo permissivo, certos caminhos críticos do sistema permanecem protegidos.

### Sandbox Desabilitado

```yaml
security:
  sandbox:
    enabled: false
```

Quando o sandboxing está desabilitado, seu assistente é executado com as mesmas permissões que o próprio processo do OpenClaw. **Isso não é recomendado para uso em produção.** Pode ser aceitável para desenvolvimento local ou testes, mas você deve estar ciente de que não há proteções.

## Ajustando Permissões do Sandbox

Às vezes o modo padrão está quase certo, mas você precisa conceder acesso a um recurso específico. Em vez de mudar todo o modo, você pode adicionar exceções direcionadas.

### Permitir Diretórios Específicos

```yaml
security:
  sandbox:
    enabled: true
    mode: "strict"
    filesystem:
      allowed_paths:
        - path: "/data/arquivos-do-projeto"
          access: "read"
        - path: "/var/openclaw/output"
          access: "read-write"
      denied_paths:
        - "/data/arquivos-do-projeto/segredos"
```

### Permitir Variáveis de Ambiente Específicas

```yaml
security:
  sandbox:
    enabled: true
    mode: "strict"
    allowed_env:
      - "NODE_ENV"
      - "API_BASE_URL"
      - "LOG_LEVEL"
```

Note que variáveis sensíveis como `OPENCLAW_API_KEY` ou `DATABASE_URL` *não* estão nesta lista. Exponha apenas o que o assistente genuinamente precisa.

### Ajustar Limites de Recursos

```yaml
security:
  sandbox:
    enabled: true
    mode: "strict"
    resource_limits:
      max_memory_mb: 512
      max_cpu_seconds: 60
      max_output_size_mb: 10
```

## Quando Relaxar o Sandboxing

Há cenários legítimos em que você pode precisar reduzir as restrições do sandbox. Aqui estão os mais comuns, junto com a abordagem recomendada para cada um.

### Seu Assistente Precisa Ler Arquivos de Projeto

**Cenário:** Você quer que seu assistente analise código, revise documentos ou processe arquivos de dados armazenados no seu servidor.

**Recomendação:** Fique no modo `strict` mas adicione o diretório específico a `allowed_paths` com acesso `read`.

```yaml
security:
  sandbox:
    enabled: true
    mode: "strict"
    filesystem:
      allowed_paths:
        - path: "/home/usuario/projetos/meu-projeto"
          access: "read"
```

### Seu Assistente Precisa Escrever Arquivos de Saída

**Cenário:** Seu assistente gera relatórios, exporta dados ou cria arquivos como parte do seu fluxo de trabalho.

**Recomendação:** Fique no modo `strict` mas adicione um diretório de saída designado com acesso `read-write`.

```yaml
filesystem:
  allowed_paths:
    - path: "/var/openclaw/output"
      access: "read-write"
```

Não dê acesso de escrita ao mesmo diretório onde seus dados de origem ficam. Mantenha entradas e saídas separadas.

### Seu Assistente Executa Ferramentas de Desenvolvimento

**Cenário:** Você está usando o OpenClaw como assistente de codificação e ele precisa executar linters, testes ou comandos de build.

**Recomendação:** Use o modo `standard` ou `permissive`, mas escope-o a uma automação ou ferramenta específica em vez de alterar a configuração global.

```yaml
security:
  sandbox:
    enabled: true
    mode: "strict"
    overrides:
      - automation: "code-review"
        mode: "standard"
        filesystem:
          allowed_paths:
            - path: "/home/usuario/projetos"
              access: "read"
      - automation: "run-tests"
        mode: "permissive"
        filesystem:
          allowed_paths:
            - path: "/home/usuario/projetos/meu-app"
              access: "read-write"
```

Dessa forma, seu sandbox global permanece estrito enquanto automações específicas obtêm o acesso de que precisam.

### Você Está Rodando Localmente para Desenvolvimento

**Cenário:** Você está desenvolvendo e testando em sua própria máquina e o atrito do sandboxing estrito está te atrasando.

**Recomendação:** É razoável usar o modo `permissive` ou até desabilitar o sandboxing durante o desenvolvimento local. Mas **nunca** leve essa configuração para produção. Use configurações específicas por ambiente:

```yaml
# openclaw.config.development.yaml
security:
  sandbox:
    enabled: true
    mode: "permissive"

# openclaw.config.production.yaml
security:
  sandbox:
    enabled: true
    mode: "strict"
```

## Sinais de Alerta

Se você se encontrar fazendo qualquer uma das seguintes coisas, pare e reconsidere:

- **Desabilitar o sandboxing em produção** porque uma automação "não funciona" com ele habilitado. A automação provavelmente precisa ser reestruturada, não o sandbox desabilitado.
- **Definir o modo como permissive globalmente** por conveniência. Use substituições por automação.
- **Adicionar `/` ou todo o seu diretório home a `allowed_paths`**. Isso derrota o propósito do sandboxing completamente.
- **Ignorar logs de violação do sandbox.** Se você vir tentativas bloqueadas no seu log de auditoria, investigue-as. Podem indicar uma configuração incorreta ou algo mais sério.

Revise tentativas bloqueadas regularmente:

```bash
openclaw audit log --filter "sandbox.blocked" --last 7d
```

```
TIMESTAMP            ACTION                  ATTEMPT                    RESULT
2026-04-19 14:22:01  assistant.analyze       read /etc/passwd           BLOCKED
2026-04-20 08:44:33  automations.export      write /tmp/data.csv        BLOCKED
2026-04-21 16:11:09  assistant.run_script    spawn process: curl        BLOCKED
```

Cada uma dessas conta uma história. A primeira pode ser uma tentativa de injeção de prompt. A segunda pode significar que você esqueceu de adicionar um diretório de saída. A terceira pode significar que seu script precisa de acesso à rede configurado no nível de política de rede em vez de gerar curl.

O sandboxing é um dos recursos de segurança mais importantes do OpenClaw. Mantenha-o habilitado, mantenha-o restrito e ajuste cirurgicamente quando necessário.
