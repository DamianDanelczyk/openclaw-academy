# Sandboxing

When your OpenClaw assistant runs an action -- reading a file, calling an API, executing a script -- you probably want to make sure it cannot accidentally (or maliciously) reach beyond what it is supposed to. That is exactly what sandboxing does. It draws a boundary around your assistant's execution environment so that actions stay contained and predictable.

Think of it like a playpen for your assistant. Inside the sandbox, it has everything it needs to do its job. Outside the sandbox, your system, files, and network are off limits.

## How Sandboxing Works

Under the hood, OpenClaw uses operating-system-level isolation to create a restricted environment for each action execution. Here is what happens when your assistant runs a tool:

1. OpenClaw creates a temporary, isolated container.
2. The action runs inside that container with limited permissions.
3. The container has access only to explicitly allowed resources (files, network endpoints, environment variables).
4. When the action finishes, the container is destroyed along with any temporary files it created.

This means that even if an action tries to do something it should not -- like reading files outside its working directory or connecting to an unexpected server -- the sandbox blocks it.

## Sandbox Modes

OpenClaw offers three sandbox modes, from most restrictive to least:

### Strict Mode (Recommended Default)

```yaml
security:
  sandbox:
    enabled: true
    mode: "strict"
```

In strict mode:
- No filesystem access outside the designated working directory
- No network access unless explicitly allowed in your network config
- No access to environment variables unless explicitly listed
- No ability to spawn child processes
- Memory and CPU time are capped

This is the right choice for most users. Start here and only relax restrictions if you have a specific reason.

### Standard Mode

```yaml
security:
  sandbox:
    enabled: true
    mode: "standard"
```

In standard mode:
- Read-only access to the filesystem (but not sensitive directories like `/etc`, home directories, or SSH keys)
- Network access follows your allowlist configuration
- Environment variables listed in `sandbox.allowed_env` are accessible
- Child processes are allowed but monitored
- Memory and CPU caps still apply

Standard mode is appropriate when your assistant needs to read project files or configuration data as part of its tasks.

### Permissive Mode

```yaml
security:
  sandbox:
    enabled: true
    mode: "permissive"
```

In permissive mode:
- Read and write access to allowed directories
- Network access follows your allowlist
- All environment variables are accessible
- Child processes are allowed
- Higher memory and CPU caps

Use permissive mode only when your assistant needs to write files, install packages, or perform development tasks. Even in permissive mode, certain critical system paths remain protected.

### Sandbox Disabled

```yaml
security:
  sandbox:
    enabled: false
```

When sandboxing is disabled, your assistant runs with the same permissions as the OpenClaw process itself. **This is not recommended for production use.** It may be acceptable for local development or testing, but you should be aware that there are no guardrails.

## Fine-Tuning Sandbox Permissions

Sometimes the default mode is almost right, but you need to grant access to one specific resource. Instead of changing the entire mode, you can add targeted exceptions.

### Allow Specific Directories

```yaml
security:
  sandbox:
    enabled: true
    mode: "strict"
    filesystem:
      allowed_paths:
        - path: "/data/project-files"
          access: "read"
        - path: "/var/openclaw/output"
          access: "read-write"
      denied_paths:
        - "/data/project-files/secrets"
```

### Allow Specific Environment Variables

```yaml
security:
  sandbox:
    enabled: true
    mode: "strict"
    allowed_env:
      - "NODE_ENV"
      - "API_BASE_URL"
      - "LOG_LEVEL"
```

Notice that sensitive variables like `OPENCLAW_API_KEY` or `DATABASE_URL` are *not* on this list. Only expose what the assistant genuinely needs.

### Adjust Resource Limits

```yaml
security:
  sandbox:
    enabled: true
    mode: "strict"
    resource_limits:
      max_memory_mb: 512
      max_cpu_seconds: 60
      max_output_size_mb: 10
```

## When to Relax Sandboxing

There are legitimate scenarios where you may need to reduce sandbox restrictions. Here are the most common ones, along with the recommended approach for each.

### Your Assistant Needs to Read Project Files

**Scenario:** You want your assistant to analyze code, review documents, or process data files stored on your server.

**Recommendation:** Stay in `strict` mode but add the specific directory to `allowed_paths` with `read` access.

```yaml
security:
  sandbox:
    enabled: true
    mode: "strict"
    filesystem:
      allowed_paths:
        - path: "/home/user/projects/my-project"
          access: "read"
```

### Your Assistant Needs to Write Output Files

**Scenario:** Your assistant generates reports, exports data, or creates files as part of its workflow.

**Recommendation:** Stay in `strict` mode but add a designated output directory with `read-write` access.

```yaml
filesystem:
  allowed_paths:
    - path: "/var/openclaw/output"
      access: "read-write"
```

Do not give write access to the same directory where your source data lives. Keep inputs and outputs separate.

### Your Assistant Runs Development Tools

**Scenario:** You are using OpenClaw as a coding assistant and it needs to run linters, tests, or build commands.

**Recommendation:** Use `standard` or `permissive` mode, but scope it to a specific automation or tool rather than changing the global setting.

```yaml
security:
  sandbox:
    enabled: true
    mode: "strict"
    overrides:
      - automation: "code-review"
        mode: "standard"
        filesystem:
          allowed_paths:
            - path: "/home/user/projects"
              access: "read"
      - automation: "run-tests"
        mode: "permissive"
        filesystem:
          allowed_paths:
            - path: "/home/user/projects/my-app"
              access: "read-write"
```

This way, your global sandbox remains strict while specific automations get the access they need.

### You Are Running Locally for Development

**Scenario:** You are developing and testing on your own machine and the friction of strict sandboxing slows you down.

**Recommendation:** It is reasonable to use `permissive` mode or even disable sandboxing during local development. But **never** carry that setting into production. Use environment-specific configs:

```yaml
# openclaw.config.development.yaml
security:
  sandbox:
    enabled: true
    mode: "permissive"

# openclaw.config.production.yaml
security:
  sandbox:
    enabled: true
    mode: "strict"
```

## Warning Signs

If you find yourself doing any of the following, stop and reconsider:

- **Disabling sandboxing in production** because an automation "does not work" with it enabled. The automation probably needs to be restructured, not the sandbox disabled.
- **Setting mode to permissive globally** for convenience. Use per-automation overrides instead.
- **Adding `/` or your entire home directory to `allowed_paths`**. This defeats the purpose of sandboxing entirely.
- **Ignoring sandbox violation logs.** If you see blocked attempts in your audit log, investigate them. They could indicate a misconfiguration or something more serious.

Review blocked attempts regularly:

```bash
openclaw audit log --filter "sandbox.blocked" --last 7d
```

```
TIMESTAMP            ACTION                  ATTEMPT                    RESULT
2026-04-19 14:22:01  assistant.analyze       read /etc/passwd           BLOCKED
2026-04-20 08:44:33  automations.export      write /tmp/data.csv        BLOCKED
2026-04-21 16:11:09  assistant.run_script    spawn process: curl        BLOCKED
```

Each of these tells a story. The first one might be a prompt injection attempt. The second might mean you forgot to add an output directory. The third might mean your script needs network access configured at the network policy level rather than by spawning curl.

Sandboxing is one of the most important security features in OpenClaw. Keep it enabled, keep it tight, and adjust surgically when needed.
