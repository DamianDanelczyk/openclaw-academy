# Tool-Richtlinien

Dein OpenClaw-Assistent hat Zugriff auf eine breite Palette von Tools — Dateien lesen, E-Mails senden, APIs aufrufen, Datenbanken verwalten und vieles mehr. Tool-Richtlinien ermöglichen dir zu entscheiden, welche dieser Tools dein Assistent tatsächlich verwenden darf und unter welchen Bedingungen.

Stell dir Tool-Richtlinien wie ein Berechtigungssystem vor. Genauso wie du nicht jedem Mitarbeiter einen Generalschlüssel für jeden Raum im Gebäude geben würdest, solltest du deinem Assistenten keinen uneingeschränkten Zugriff auf alle Tools geben. Das Prinzip ist einfach: Gewähre Zugriff auf das, was benötigt wird, und schränke alles andere ein.

## Wie Tool-Richtlinien funktionieren

Jedes Tool in OpenClaw hat einen Namen, der in einer punktgetrennten Hierarchie organisiert ist:

```
calendar.get_events
calendar.create_event
calendar.delete_event
email.read
email.send
email.delete
system.backup
system.delete_files
system.execute_command
github.get_issues
github.create_pull_request
```

Tool-Richtlinien weisen jedem Tool oder jeder Gruppe von Tools eine von drei Zugangsstufen zu:

| Richtlinie | Was sie bewirkt |
|------------|-----------------|
| `allow` | Das Tool wird sofort ausgeführt, ohne nachzufragen |
| `ask` | OpenClaw hält inne und bittet dich um Genehmigung, bevor es ausgeführt wird |
| `deny` | Das Tool ist vollständig gesperrt. Der Assistent kann es nicht verwenden. |

## Grundlegende Konfiguration

Tool-Richtlinien befinden sich in deiner `openclaw.config.yaml` im `security`-Abschnitt:

```yaml
security:
  tool_policies:
    default: "deny"
    rules:
      - tools: ["calendar.*"]
        policy: "allow"
      - tools: ["email.read"]
        policy: "allow"
      - tools: ["email.send"]
        policy: "ask"
      - tools: ["email.delete"]
        policy: "deny"
```

Lass uns diese Konfiguration laut lesen:

- **Standard: deny.** Wenn ein Tool in keiner Regel erwähnt wird, ist es gesperrt.
- **Alle Kalender-Tools sind erlaubt.** Der Assistent kann Kalendertermine frei lesen, erstellen und ändern.
- **E-Mails lesen ist erlaubt.** Der Assistent kann deinen Posteingang prüfen, ohne nachzufragen.
- **E-Mails senden erfordert Genehmigung.** Der Assistent zeigt dir den Entwurf und wartet auf dein OK.
- **E-Mails löschen ist gesperrt.** Der Assistent kann keine E-Mails löschen, basta.

### Die Bedeutung von Standard-Deny

Die Einstellung `default: "deny"` wird dringend empfohlen. Sie bedeutet, dass jedes neue Tool, das zu OpenClaw hinzugefügt wird, standardmäßig gesperrt ist, bis du es explizit freigibst. Das ist viel sicherer als `default: "allow"`, bei dem jedes neue Tool automatisch zugänglich ist.

```yaml
# Sicher: neue Tools sind standardmäßig gesperrt
security:
  tool_policies:
    default: "deny"

# Riskant: neue Tools sind standardmäßig zugänglich
security:
  tool_policies:
    default: "allow"
```

## Wildcard-Muster

Du kannst `*` als Platzhalter verwenden, um Gruppen von Tools zu treffen:

| Muster | Trifft |
|--------|--------|
| `calendar.*` | Alle Kalender-Tools |
| `email.*` | Alle E-Mail-Tools |
| `system.*` | Alle System-Tools |
| `*.read` | Jedes Tool, das auf `.read` endet |
| `*` | Alles |

Du kannst auch mehrere spezifische Tools in einer Regel auflisten:

```yaml
- tools: ["weather.get_forecast", "news.get_headlines", "calendar.get_events"]
  policy: "allow"
```

## Bedingte Richtlinien

Manchmal möchtest du, dass ein Tool unter bestimmten Umständen erlaubt und unter anderen gesperrt ist. Bedingte Richtlinien ermöglichen es dir, Kontext zu deinen Regeln hinzuzufügen:

### Zeitbasierte Einschränkungen

```yaml
rules:
  - tools: ["email.send"]
    policy: "allow"
    conditions:
      time_window:
        start: "09:00"
        end: "18:00"
        timezone: "America/New_York"
        days: ["mon", "tue", "wed", "thu", "fri"]
    fallback_policy: "deny"
```

Das erlaubt dem Assistenten, während der Geschäftszeiten E-Mails zu senden, sperrt ihn aber außerhalb dieser Zeiten. Nützlich, um versehentliche 3-Uhr-E-Mails zu verhindern.

### Quellenbasierte Einschränkungen

```yaml
rules:
  - tools: ["system.execute_command"]
    policy: "allow"
    conditions:
      triggered_by: ["automation:code-review"]
    fallback_policy: "deny"
```

Das erlaubt `system.execute_command` nur, wenn es von der `code-review`-Automation ausgelöst wird. Wenn etwas anderes es versucht, wird es abgelehnt.

### Argumentbasierte Einschränkungen

```yaml
rules:
  - tools: ["github.create_pull_request"]
    policy: "allow"
    conditions:
      arguments:
        repo:
          pattern: "my-org/*"
    fallback_policy: "ask"
```

Der Assistent kann PRs in Repositories unter `my-org` ohne Nachfrage erstellen, aber wird für jedes andere Repository um Genehmigung bitten.

## Empfohlene Richtlinien nach Anwendungsfall

Hier sind vollständige Richtlinienkonfigurationen für häufige Szenarien. Kopiere diejenige, die zu deiner Situation passt, und passe sie nach Bedarf an.

### Persönlicher Assistent

Für einen Allzweck-Assistenten, der deinen Kalender, E-Mails, Notizen und Informationssuchen verwaltet:

```yaml
security:
  tool_policies:
    default: "deny"
    rules:
      # Informationsabfrage — sicher, frei zu erlauben
      - tools: ["calendar.get_events", "weather.*", "news.*", "search.*"]
        policy: "allow"

      # Inhalte erstellen — mit etwas Aufsicht erlauben
      - tools: ["calendar.create_event", "calendar.update_event"]
        policy: "allow"
      - tools: ["notes.create", "notes.update"]
        policy: "allow"

      # Kommunikation — Genehmigung erforderlich
      - tools: ["email.read"]
        policy: "allow"
      - tools: ["email.send", "email.reply"]
        policy: "ask"
      - tools: ["slack.send_message"]
        policy: "ask"

      # Destruktive Aktionen — vollständig sperren
      - tools: ["*.delete*", "system.*"]
        policy: "deny"
```

### Entwicklungsassistent

Für einen Coding-Assistenten, der bei Code-Reviews, Tests und Projektmanagement hilft:

```yaml
security:
  tool_policies:
    default: "deny"
    rules:
      # Code lesen und analysieren
      - tools: ["filesystem.read", "filesystem.list", "filesystem.search"]
        policy: "allow"

      # Code schreiben — in Projektverzeichnissen erlauben
      - tools: ["filesystem.write", "filesystem.create"]
        policy: "allow"
        conditions:
          arguments:
            path:
              pattern: "/home/user/projects/*"
        fallback_policy: "ask"

      # Entwicklungstools
      - tools: ["git.*", "github.get_*", "github.list_*"]
        policy: "allow"
      - tools: ["github.create_pull_request", "github.create_issue"]
        policy: "ask"
      - tools: ["system.execute_command"]
        policy: "ask"

      # Paketverwaltung
      - tools: ["npm.install", "pip.install"]
        policy: "ask"

      # Gefährliche Operationen
      - tools: ["git.force_push", "git.reset_hard", "system.delete_*"]
        policy: "deny"
```

### Kundensupport-Bot

Für einen Assistenten, der Kundenanfragen mit Zugriff auf eine Wissensdatenbank und ein Ticketsystem bearbeitet:

```yaml
security:
  tool_policies:
    default: "deny"
    rules:
      # Wissensdatenbank — nur Lesen
      - tools: ["kb.search", "kb.get_article"]
        policy: "allow"

      # Ticketing
      - tools: ["tickets.get", "tickets.list", "tickets.search"]
        policy: "allow"
      - tools: ["tickets.create", "tickets.update", "tickets.add_comment"]
        policy: "allow"
      - tools: ["tickets.close", "tickets.assign"]
        policy: "ask"
      - tools: ["tickets.delete"]
        policy: "deny"

      # Kundendaten — mit Einschränkungen lesen
      - tools: ["customers.get_profile"]
        policy: "allow"
      - tools: ["customers.update_profile"]
        policy: "ask"
      - tools: ["customers.delete_account"]
        policy: "deny"

      # Kommunikation
      - tools: ["chat.send_reply"]
        policy: "allow"
      - tools: ["email.send"]
        policy: "ask"
```

### Datenpipeline-Automation

Für einen Assistenten, der Datenverarbeitungsworkflows verwaltet:

```yaml
security:
  tool_policies:
    default: "deny"
    rules:
      # Daten lesen
      - tools: ["database.query", "database.list_tables"]
        policy: "allow"
      - tools: ["filesystem.read"]
        policy: "allow"
        conditions:
          arguments:
            path:
              pattern: "/data/*"

      # Daten schreiben — kontrolliert
      - tools: ["database.insert", "database.update"]
        policy: "ask"
      - tools: ["filesystem.write"]
        policy: "ask"
        conditions:
          arguments:
            path:
              pattern: "/data/output/*"

      # Destruktive Operationen niemals erlauben
      - tools: ["database.drop_*", "database.truncate", "database.delete"]
        policy: "deny"
      - tools: ["filesystem.delete"]
        policy: "deny"

      # Reporting
      - tools: ["reports.generate", "reports.export"]
        policy: "allow"
      - tools: ["email.send"]
        policy: "ask"
```

## Richtlinienpriorität und Konflikte

Wenn mehrere Regeln auf dasselbe Tool passen, verwendet OpenClaw folgende Regeln zur Konfliktlösung:

1. **Spezifischere Regeln gewinnen.** `email.send` schlägt `email.*`, das `*` schlägt.
2. **Explizites Deny gewinnt über Allow.** Wenn eine Regel `allow` und eine andere `deny` für dasselbe Tool sagt, gewinnt `deny`.
3. **Bedingte Regeln werden der Reihe nach ausgewertet.** Die erste passende bedingte Regel greift.

Wenn du dir jemals unsicher bist, welche Regel gelten würde, verwende `openclaw policy test` zur Prüfung.

## Richtlinien ohne Ausfallzeit aktualisieren

Nachdem du deine `openclaw.config.yaml` bearbeitet hast, lade die Richtlinien neu:

```bash
openclaw policy reload
```

```
Policies reloaded.
  Total rules: 12
  Default policy: deny
  Tools allowed: 8
  Tools requiring approval: 3
  Tools denied: 14
```

Das Neuladen erfolgt sofort und unterbricht keine laufenden Automationen. Neue Richtlinienregeln gelten für den nächsten Tool-Aufruf.
