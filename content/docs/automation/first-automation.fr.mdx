# Votre Première Automatisation

Lire sur les automatisations est une chose. Les construire, c'est là que le vrai apprentissage se passe. Dans ce guide, vous créerez trois automatisations complètes de A à Z. Chacune s'appuie sur la précédente, de sorte qu'à la fin vous aurez une bonne compréhension de la façon dont les automatisations OpenClaw fonctionnent en pratique.

Aucune expérience préalable n'est requise. Chaque étape est montrée, chaque configuration est incluse et chaque décision est expliquée.

## Automatisation 1 : Le Briefing Matinal Quotidien

C'est la classique automatisation de démarrage. Chaque jour ouvrable à 7h30, OpenClaw compilera un résumé de vos événements de calendrier, des prévisions météo et des principaux titres d'actualité, puis vous le livrera en un seul message.

### Étape 1 : Définir le Fichier d'Automatisation

Créez un nouveau fichier à `automations/morning-briefing.yaml` :

```yaml
name: "morning-briefing"
description: "Delivers a personalized morning summary on weekdays"
version: 1

trigger:
  type: "cron"
  cron: "30 7 * * 1-5"
  timezone: "America/New_York"

steps:
  - id: "fetch-calendar"
    action: "calendar.get_events"
    params:
      date: "today"
      max_results: 10
    output: "calendar_events"

  - id: "fetch-weather"
    action: "weather.get_forecast"
    params:
      location: "auto"
      period: "today"
    output: "weather"

  - id: "fetch-news"
    action: "news.get_headlines"
    params:
      categories: ["technology", "business"]
      max_results: 5
    output: "headlines"

  - id: "compile-and-send"
    action: "assistant.send_message"
    params:
      template: |
        Good morning! Here is your briefing for {{ today | date_format }}.

        ☀️ Weather: {{ weather.summary }}, {{ weather.high }}° / {{ weather.low }}°

        📅 Calendar ({{ calendar_events | length }} events):
        {% for event in calendar_events %}
        • {{ event.time }} — {{ event.title }}
        {% endfor %}

        📰 Headlines:
        {% for article in headlines %}
        • {{ article.title }} ({{ article.source }})
        {% endfor %}

        Have a great day!
```

### Étape 2 : L'Enregistrer

```bash
openclaw automation add automations/morning-briefing.yaml
```

```
✓ Automation "morning-briefing" registered
  Next run: Mon Apr 27 07:30 (America/New_York)
```

### Étape 3 : La Tester Maintenant

Vous n'avez pas à attendre demain matin :

```bash
openclaw automation run "morning-briefing" --now
```

Cela déclenche les quatre étapes immédiatement. Vous verrez une sortie comme celle-ci :

```
[Step 1/4] fetch-calendar ............ ✓ (3 events found)
[Step 2/4] fetch-weather ............. ✓ (Partly cloudy, 72°/58°)
[Step 3/4] fetch-news ................ ✓ (5 headlines fetched)
[Step 4/4] compile-and-send .......... ✓ (Message delivered)

Automation "morning-briefing" completed in 2.4s
```

### Étape 4 : La Personnaliser

Vous voulez ajouter une étape ? Insérez-la n'importe où dans la liste `steps`. Par exemple, pour inclure votre liste de tâches :

```yaml
  - id: "fetch-tasks"
    action: "tasks.get_due"
    params:
      date: "today"
    output: "due_tasks"
```

Puis référencez `{{ due_tasks }}` dans votre template. Le moteur d'automatisation exécute les étapes dans l'ordre, et la sortie de chaque étape est disponible pour toutes les étapes suivantes.

## Automatisation 2 : Répondeur d'E-mails Intelligent

Celle-ci surveille les e-mails entrants qui correspondent à certains critères et rédige une réponse. Elle n'enverra pas automatiquement — vous approuverez chaque brouillon avant qu'il ne parte.

### Étape 1 : Créer le Fichier d'Automatisation

Créez `automations/email-responder.yaml` :

```yaml
name: "email-responder"
description: "Drafts replies for specific incoming emails"
version: 1

trigger:
  type: "webhook"
  path: "/hooks/incoming-email"
  secret: "${EMAIL_WEBHOOK_SECRET}"
  signature_header: "X-Mailgun-Signature"
  signature_algorithm: "sha256"

filter:
  conditions:
    - field: "payload.sender"
      operator: "ends_with"
      value: "@clients.example.com"
    - field: "payload.subject"
      operator: "not_contains"
      value: "URGENT"

steps:
  - id: "analyze-email"
    action: "assistant.analyze"
    params:
      input: "{{ payload.body }}"
      instructions: "Classify this email as one of: question, feedback, scheduling, other"
    output: "analysis"

  - id: "draft-reply"
    action: "assistant.compose"
    params:
      context:
        original_email: "{{ payload.body }}"
        sender: "{{ payload.sender }}"
        category: "{{ analysis.category }}"
      instructions: |
        Draft a professional, friendly reply to this email.
        Keep it concise. If it is a scheduling request, suggest
        three available time slots from my calendar.
      tone: "professional"
    output: "draft"

  - id: "save-draft"
    action: "email.save_draft"
    params:
      to: "{{ payload.sender }}"
      subject: "Re: {{ payload.subject }}"
      body: "{{ draft.text }}"
      in_reply_to: "{{ payload.message_id }}"
    output: "saved_draft"

  - id: "notify-me"
    action: "assistant.send_message"
    params:
      template: |
        New draft reply ready for review.

        From: {{ payload.sender }}
        Subject: {{ payload.subject }}
        Category: {{ analysis.category }}

        Draft preview: {{ draft.text | truncate(200) }}

        👉 Review and send: {{ saved_draft.review_url }}
```

### Étape 2 : Configurer le Secret Webhook

Ajoutez le secret à votre fichier `.env` :

```bash
# .env
EMAIL_WEBHOOK_SECRET=whsec_em_4f8a2c6d9e1b3f5a7e9c2b4d6f8a0e1c
```

### Étape 3 : Enregistrer et Connecter

```bash
openclaw automation add automations/email-responder.yaml
openclaw config reload
```

Maintenant configurez votre fournisseur d'e-mail (Mailgun, SendGrid, etc.) pour transférer les e-mails entrants à :

```
https://your-instance.openclaw.app/hooks/incoming-email
```

### Étape 4 : Tester avec un E-mail Simulé

```bash
openclaw automation run "email-responder" --now --payload '{
  "sender": "alice@clients.example.com",
  "subject": "Meeting next week?",
  "body": "Hi, could we schedule a call sometime next week to discuss the project? Tuesday or Wednesday would work best for me.",
  "message_id": "msg-test-001"
}'
```

```
[Step 1/4] analyze-email ............ ✓ (category: scheduling)
[Step 2/4] draft-reply .............. ✓ (247 words)
[Step 3/4] save-draft ............... ✓ (draft saved)
[Step 4/4] notify-me ................ ✓ (notification sent)
```

Le bloc de filtrage garde les choses concentrées : seuls les e-mails de `@clients.example.com` qui ne sont *pas* marqués URGENT déclenchent cette automatisation. Tout le reste est ignoré.

## Automatisation 3 : Moniteur de Santé de Projet

Cette automatisation s'exécute toutes les quatre heures pendant les jours ouvrables. Elle vérifie plusieurs métriques d'un projet -- les issues ouvertes, l'état de la build, la disponibilité -- et envoie une alerte uniquement si quelque chose semble anormal. Pas d'alerte signifie que tout va bien.

### Étape 1 : Créer le Fichier d'Automatisation

Créez `automations/project-monitor.yaml` :

```yaml
name: "project-monitor"
description: "Checks project health metrics and alerts on problems"
version: 1

trigger:
  type: "cron"
  cron: "0 8,12,16,20 * * 1-5"
  timezone: "UTC"

config:
  project_repo: "your-org/main-project"
  uptime_url: "https://api.example.com/health"
  alert_channel: "slack"

steps:
  - id: "check-issues"
    action: "github.get_issues"
    params:
      repo: "{{ config.project_repo }}"
      state: "open"
      labels: ["bug", "critical"]
    output: "critical_issues"

  - id: "check-build"
    action: "github.get_workflow_status"
    params:
      repo: "{{ config.project_repo }}"
      workflow: "ci.yml"
      branch: "main"
    output: "build_status"

  - id: "check-uptime"
    action: "http.get"
    params:
      url: "{{ config.uptime_url }}"
      timeout: 10
    output: "uptime_check"

  - id: "evaluate-health"
    action: "assistant.analyze"
    params:
      input:
        critical_issues: "{{ critical_issues }}"
        build: "{{ build_status }}"
        uptime: "{{ uptime_check }}"
      instructions: |
        Evaluate the project health based on these inputs.
        Flag as "unhealthy" if any of these are true:
        - More than 3 critical/bug issues are open
        - The latest CI build failed
        - The uptime check returned a non-200 status
        Otherwise, flag as "healthy".
    output: "health_report"

  - id: "alert-if-needed"
    action: "assistant.send_message"
    condition: "health_report.status == 'unhealthy'"
    params:
      channel: "{{ config.alert_channel }}"
      template: |
        ⚠️ Project Health Alert — {{ now | date_format }}

        Status: {{ health_report.status | upper }}

        {% if critical_issues | length > 3 %}
        🐛 {{ critical_issues | length }} critical issues open (threshold: 3)
        {% endif %}

        {% if build_status.conclusion != 'success' %}
        🔴 Latest CI build: {{ build_status.conclusion }}
           Run: {{ build_status.html_url }}
        {% endif %}

        {% if uptime_check.status_code != 200 %}
        🌐 Uptime check failed (HTTP {{ uptime_check.status_code }})
        {% endif %}

        {{ health_report.summary }}
```

### Étape 2 : L'Enregistrer

```bash
openclaw automation add automations/project-monitor.yaml
```

### Étape 3 : Vérifier Toutes les Connexions

```bash
openclaw automation check "project-monitor"
```

```
Checking automation "project-monitor"...
  ✓ GitHub token valid (repo: your-org/main-project)
  ✓ HTTP endpoint reachable (https://api.example.com/health)
  ✓ Slack channel connected
  ✓ Cron schedule valid (next: Today 16:00 UTC)

All checks passed.
```

### Étape 4 : Faire une Simulation

```bash
openclaw automation run "project-monitor" --now --dry-run
```

Le flag `--dry-run` exécute toutes les étapes mais n'envoie pas réellement l'alerte. Il vous montre ce qui *se serait* passé :

```
[Step 1/5] check-issues ............. ✓ (2 critical issues)
[Step 2/5] check-build .............. ✓ (conclusion: success)
[Step 3/5] check-uptime ............. ✓ (HTTP 200)
[Step 4/5] evaluate-health .......... ✓ (status: healthy)
[Step 5/5] alert-if-needed .......... ⏭ SKIPPED (condition not met)

DRY RUN: No messages sent. Project is healthy.
```

Notez le champ `condition` sur la dernière étape. L'alerte ne se déclenche que quand les choses ne vont pas bien. Cela signifie que vous ne serez pas inondé de messages "tout va bien" quatre fois par jour.

## Ce Que Vous Avez Appris

À travers ces trois automatisations, vous avez utilisé :

- **Des déclencheurs cron** pour la planification basée sur le temps
- **Des déclencheurs webhook** pour les réponses pilotées par les événements
- **Des workflows multi-étapes** avec des données transmises entre les étapes
- **De la logique conditionnelle** pour envoyer des alertes uniquement quand nécessaire
- **Des filtres** pour préciser quels événements déclenchent l'automatisation
- **Des templates** pour une sortie lisible et dynamique
- **Des simulations** et commandes de test pour un développement sécurisé

Chaque automatisation dans OpenClaw suit cette même structure : déclencheur, filtre optionnel, étapes, conditions optionnelles. Une fois à l'aise avec ces éléments de base, vous pouvez les combiner pour gérer pratiquement n'importe quel flux de travail que vous pouvez imaginer.
