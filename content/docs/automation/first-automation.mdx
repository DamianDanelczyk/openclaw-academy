# Your First Automation

Reading about automations is one thing. Building them is where the real learning happens. In this guide, you will create three complete automations from start to finish. Each one builds on the last, so by the end you will have a solid feel for how OpenClaw automations work in practice.

No prior experience is required. Every step is shown, every config is included, and every decision is explained.

## Automation 1: The Daily Morning Briefing

This is the classic starter automation. Every weekday at 7:30 AM, OpenClaw will compile a summary of your calendar events, the weather forecast, and top news headlines, then deliver it to you as a single message.

### Step 1: Define the Automation File

Create a new file at `automations/morning-briefing.yaml`:

```yaml
name: "morning-briefing"
description: "Delivers a personalized morning summary on weekdays"
version: 1

trigger:
  type: "cron"
  cron: "30 7 * * 1-5"
  timezone: "America/New_York"

steps:
  - id: "fetch-calendar"
    action: "calendar.get_events"
    params:
      date: "today"
      max_results: 10
    output: "calendar_events"

  - id: "fetch-weather"
    action: "weather.get_forecast"
    params:
      location: "auto"
      period: "today"
    output: "weather"

  - id: "fetch-news"
    action: "news.get_headlines"
    params:
      categories: ["technology", "business"]
      max_results: 5
    output: "headlines"

  - id: "compile-and-send"
    action: "assistant.send_message"
    params:
      template: |
        Good morning! Here is your briefing for {{ today | date_format }}.

        ☀️ Weather: {{ weather.summary }}, {{ weather.high }}° / {{ weather.low }}°

        📅 Calendar ({{ calendar_events | length }} events):
        {% for event in calendar_events %}
        • {{ event.time }} — {{ event.title }}
        {% endfor %}

        📰 Headlines:
        {% for article in headlines %}
        • {{ article.title }} ({{ article.source }})
        {% endfor %}

        Have a great day!
```

### Step 2: Register It

```bash
openclaw automation add automations/morning-briefing.yaml
```

```
✓ Automation "morning-briefing" registered
  Next run: Mon Apr 27 07:30 (America/New_York)
```

### Step 3: Test It Right Now

You do not have to wait until tomorrow morning:

```bash
openclaw automation run "morning-briefing" --now
```

This triggers all four steps immediately. You will see output like this:

```
[Step 1/4] fetch-calendar ............ ✓ (3 events found)
[Step 2/4] fetch-weather ............. ✓ (Partly cloudy, 72°/58°)
[Step 3/4] fetch-news ................ ✓ (5 headlines fetched)
[Step 4/4] compile-and-send .......... ✓ (Message delivered)

Automation "morning-briefing" completed in 2.4s
```

### Step 4: Customize It

Want to add a step? Insert it anywhere in the `steps` list. For example, to include your task list:

```yaml
  - id: "fetch-tasks"
    action: "tasks.get_due"
    params:
      date: "today"
    output: "due_tasks"
```

Then reference `{{ due_tasks }}` in your template. The automation engine runs steps in order, and each step's output is available to all subsequent steps.

## Automation 2: Smart Email Responder

This one watches for incoming emails that match certain criteria and drafts a reply. It will not send automatically -- you will approve each draft before it goes out.

### Step 1: Create the Automation File

Create `automations/email-responder.yaml`:

```yaml
name: "email-responder"
description: "Drafts replies for specific incoming emails"
version: 1

trigger:
  type: "webhook"
  path: "/hooks/incoming-email"
  secret: "${EMAIL_WEBHOOK_SECRET}"
  signature_header: "X-Mailgun-Signature"
  signature_algorithm: "sha256"

filter:
  conditions:
    - field: "payload.sender"
      operator: "ends_with"
      value: "@clients.example.com"
    - field: "payload.subject"
      operator: "not_contains"
      value: "URGENT"

steps:
  - id: "analyze-email"
    action: "assistant.analyze"
    params:
      input: "{{ payload.body }}"
      instructions: "Classify this email as one of: question, feedback, scheduling, other"
    output: "analysis"

  - id: "draft-reply"
    action: "assistant.compose"
    params:
      context:
        original_email: "{{ payload.body }}"
        sender: "{{ payload.sender }}"
        category: "{{ analysis.category }}"
      instructions: |
        Draft a professional, friendly reply to this email.
        Keep it concise. If it is a scheduling request, suggest
        three available time slots from my calendar.
      tone: "professional"
    output: "draft"

  - id: "save-draft"
    action: "email.save_draft"
    params:
      to: "{{ payload.sender }}"
      subject: "Re: {{ payload.subject }}"
      body: "{{ draft.text }}"
      in_reply_to: "{{ payload.message_id }}"
    output: "saved_draft"

  - id: "notify-me"
    action: "assistant.send_message"
    params:
      template: |
        New draft reply ready for review.

        From: {{ payload.sender }}
        Subject: {{ payload.subject }}
        Category: {{ analysis.category }}

        Draft preview: {{ draft.text | truncate(200) }}

        👉 Review and send: {{ saved_draft.review_url }}
```

### Step 2: Set Up the Webhook Secret

Add the secret to your `.env` file:

```bash
# .env
EMAIL_WEBHOOK_SECRET=whsec_em_4f8a2c6d9e1b3f5a7c
```

### Step 3: Register and Connect

```bash
openclaw automation add automations/email-responder.yaml
openclaw config reload
```

Now configure your email provider (Mailgun, SendGrid, etc.) to forward incoming emails to:

```
https://your-instance.openclaw.app/hooks/incoming-email
```

### Step 4: Test with a Simulated Email

```bash
openclaw automation run "email-responder" --now --payload '{
  "sender": "alice@clients.example.com",
  "subject": "Meeting next week?",
  "body": "Hi, could we schedule a call sometime next week to discuss the project? Tuesday or Wednesday would work best for me.",
  "message_id": "msg-test-001"
}'
```

```
[Step 1/4] analyze-email ............ ✓ (category: scheduling)
[Step 2/4] draft-reply .............. ✓ (247 words)
[Step 3/4] save-draft ............... ✓ (draft saved)
[Step 4/4] notify-me ................ ✓ (notification sent)
```

The filter block keeps things focused: only emails from `@clients.example.com` that are *not* marked URGENT trigger this automation. Everything else is ignored.

## Automation 3: Project Health Monitor

This automation runs every four hours during business days. It checks several metrics about a project -- open issues, build status, uptime -- and sends an alert only if something looks off. No alert means everything is fine.

### Step 1: Create the Automation File

Create `automations/project-monitor.yaml`:

```yaml
name: "project-monitor"
description: "Checks project health metrics and alerts on problems"
version: 1

trigger:
  type: "cron"
  cron: "0 8,12,16,20 * * 1-5"
  timezone: "UTC"

config:
  project_repo: "your-org/main-project"
  uptime_url: "https://api.example.com/health"
  alert_channel: "slack"

steps:
  - id: "check-issues"
    action: "github.get_issues"
    params:
      repo: "{{ config.project_repo }}"
      state: "open"
      labels: ["bug", "critical"]
    output: "critical_issues"

  - id: "check-build"
    action: "github.get_workflow_status"
    params:
      repo: "{{ config.project_repo }}"
      workflow: "ci.yml"
      branch: "main"
    output: "build_status"

  - id: "check-uptime"
    action: "http.get"
    params:
      url: "{{ config.uptime_url }}"
      timeout: 10
    output: "uptime_check"

  - id: "evaluate-health"
    action: "assistant.analyze"
    params:
      input:
        critical_issues: "{{ critical_issues }}"
        build: "{{ build_status }}"
        uptime: "{{ uptime_check }}"
      instructions: |
        Evaluate the project health based on these inputs.
        Flag as "unhealthy" if any of these are true:
        - More than 3 critical/bug issues are open
        - The latest CI build failed
        - The uptime check returned a non-200 status
        Otherwise, flag as "healthy".
    output: "health_report"

  - id: "alert-if-needed"
    action: "assistant.send_message"
    condition: "health_report.status == 'unhealthy'"
    params:
      channel: "{{ config.alert_channel }}"
      template: |
        ⚠️ Project Health Alert — {{ now | date_format }}

        Status: {{ health_report.status | upper }}

        {% if critical_issues | length > 3 %}
        🐛 {{ critical_issues | length }} critical issues open (threshold: 3)
        {% endif %}

        {% if build_status.conclusion != 'success' %}
        🔴 Latest CI build: {{ build_status.conclusion }}
           Run: {{ build_status.html_url }}
        {% endif %}

        {% if uptime_check.status_code != 200 %}
        🌐 Uptime check failed (HTTP {{ uptime_check.status_code }})
        {% endif %}

        {{ health_report.summary }}
```

### Step 2: Register It

```bash
openclaw automation add automations/project-monitor.yaml
```

### Step 3: Verify All Connections

```bash
openclaw automation check "project-monitor"
```

```
Checking automation "project-monitor"...
  ✓ GitHub token valid (repo: your-org/main-project)
  ✓ HTTP endpoint reachable (https://api.example.com/health)
  ✓ Slack channel connected
  ✓ Cron schedule valid (next: Today 16:00 UTC)

All checks passed.
```

### Step 4: Do a Dry Run

```bash
openclaw automation run "project-monitor" --now --dry-run
```

The `--dry-run` flag executes all steps but does not actually send the alert. It shows you what *would* have happened:

```
[Step 1/5] check-issues ............. ✓ (2 critical issues)
[Step 2/5] check-build .............. ✓ (conclusion: success)
[Step 3/5] check-uptime ............. ✓ (HTTP 200)
[Step 4/5] evaluate-health .......... ✓ (status: healthy)
[Step 5/5] alert-if-needed .......... ⏭ SKIPPED (condition not met)

DRY RUN: No messages sent. Project is healthy.
```

Notice the `condition` field on the last step. The alert only fires when things are unhealthy. This means you will not get spammed with "everything is fine" messages four times a day.

## What You Have Learned

Across these three automations, you have used:

- **Cron triggers** for time-based scheduling
- **Webhook triggers** for event-driven responses
- **Multi-step workflows** with data passed between steps
- **Conditional logic** to send alerts only when needed
- **Filters** to narrow which events trigger the automation
- **Templates** for readable, dynamic output
- **Dry runs** and test commands for safe development

Every automation in OpenClaw follows this same structure: trigger, optional filter, steps, optional conditions. Once you are comfortable with these building blocks, you can combine them to handle virtually any workflow you can think of.
