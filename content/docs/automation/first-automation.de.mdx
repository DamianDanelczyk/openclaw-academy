# Deine erste Automation

Über Automationen zu lesen ist eine Sache. Sie zu bauen ist der Ort, wo das eigentliche Lernen stattfindet. In diesem Leitfaden wirst du drei vollständige Automationen von Anfang bis Ende erstellen. Jede baut auf der vorherigen auf, sodass du am Ende ein solides Gefühl dafür hast, wie OpenClaw-Automationen in der Praxis funktionieren.

Keine Vorkenntnisse erforderlich. Jeder Schritt wird gezeigt, jede Konfiguration ist enthalten, und jede Entscheidung wird erklärt.

## Automation 1: Das tägliche Morgen-Briefing

Das ist die klassische Starter-Automation. Jeden Wochentag um 7:30 Uhr wird OpenClaw eine Zusammenfassung deiner Kalendertermine, der Wettervorhersage und der wichtigsten Nachrichten-Schlagzeilen kompilieren und sie dir als eine einzige Nachricht zustellen.

### Schritt 1: Die Automationsdatei definieren

Erstelle eine neue Datei unter `automations/morning-briefing.yaml`:

```yaml
name: "morning-briefing"
description: "Delivers a personalized morning summary on weekdays"
version: 1

trigger:
  type: "cron"
  cron: "30 7 * * 1-5"
  timezone: "America/New_York"

steps:
  - id: "fetch-calendar"
    action: "calendar.get_events"
    params:
      date: "today"
      max_results: 10
    output: "calendar_events"

  - id: "fetch-weather"
    action: "weather.get_forecast"
    params:
      location: "auto"
      period: "today"
    output: "weather"

  - id: "fetch-news"
    action: "news.get_headlines"
    params:
      categories: ["technology", "business"]
      max_results: 5
    output: "headlines"

  - id: "compile-and-send"
    action: "assistant.send_message"
    params:
      template: |
        Good morning! Here is your briefing for {{ today | date_format }}.

        ☀️ Weather: {{ weather.summary }}, {{ weather.high }}° / {{ weather.low }}°

        📅 Calendar ({{ calendar_events | length }} events):
        {% for event in calendar_events %}
        • {{ event.time }} — {{ event.title }}
        {% endfor %}

        📰 Headlines:
        {% for article in headlines %}
        • {{ article.title }} ({{ article.source }})
        {% endfor %}

        Have a great day!
```

### Schritt 2: Registrieren

```bash
openclaw automation add automations/morning-briefing.yaml
```

```
✓ Automation "morning-briefing" registered
  Next run: Mon Apr 27 07:30 (America/New_York)
```

### Schritt 3: Jetzt sofort testen

Du musst nicht bis morgen früh warten:

```bash
openclaw automation run "morning-briefing" --now
```

Damit werden alle vier Schritte sofort ausgelöst. Du wirst eine Ausgabe wie diese sehen:

```
[Step 1/4] fetch-calendar ............ ✓ (3 events found)
[Step 2/4] fetch-weather ............. ✓ (Partly cloudy, 72°/58°)
[Step 3/4] fetch-news ................ ✓ (5 headlines fetched)
[Step 4/4] compile-and-send .......... ✓ (Message delivered)

Automation "morning-briefing" completed in 2.4s
```

### Schritt 4: Anpassen

Möchtest du einen Schritt hinzufügen? Füge ihn an beliebiger Stelle in der `steps`-Liste ein. Um zum Beispiel deine Aufgabenliste einzubeziehen:

```yaml
  - id: "fetch-tasks"
    action: "tasks.get_due"
    params:
      date: "today"
    output: "due_tasks"
```

Dann referenziere `{{ due_tasks }}` in deiner Vorlage. Die Automationsmaschine führt Schritte der Reihe nach aus, und die Ausgabe jedes Schritts ist für alle nachfolgenden Schritte verfügbar.

## Automation 2: Intelligenter E-Mail-Beantworter

Dieser überwacht eingehende E-Mails, die bestimmte Kriterien erfüllen, und erstellt einen Antwort-Entwurf. Er sendet nicht automatisch — du wirst jeden Entwurf genehmigen, bevor er rausgeht.

### Schritt 1: Die Automationsdatei erstellen

Erstelle `automations/email-responder.yaml`:

```yaml
name: "email-responder"
description: "Drafts replies for specific incoming emails"
version: 1

trigger:
  type: "webhook"
  path: "/hooks/incoming-email"
  secret: "${EMAIL_WEBHOOK_SECRET}"
  signature_header: "X-Mailgun-Signature"
  signature_algorithm: "sha256"

filter:
  conditions:
    - field: "payload.sender"
      operator: "ends_with"
      value: "@clients.example.com"
    - field: "payload.subject"
      operator: "not_contains"
      value: "URGENT"

steps:
  - id: "analyze-email"
    action: "assistant.analyze"
    params:
      input: "{{ payload.body }}"
      instructions: "Classify this email as one of: question, feedback, scheduling, other"
    output: "analysis"

  - id: "draft-reply"
    action: "assistant.compose"
    params:
      context:
        original_email: "{{ payload.body }}"
        sender: "{{ payload.sender }}"
        category: "{{ analysis.category }}"
      instructions: |
        Draft a professional, friendly reply to this email.
        Keep it concise. If it is a scheduling request, suggest
        three available time slots from my calendar.
      tone: "professional"
    output: "draft"

  - id: "save-draft"
    action: "email.save_draft"
    params:
      to: "{{ payload.sender }}"
      subject: "Re: {{ payload.subject }}"
      body: "{{ draft.text }}"
      in_reply_to: "{{ payload.message_id }}"
    output: "saved_draft"

  - id: "notify-me"
    action: "assistant.send_message"
    params:
      template: |
        New draft reply ready for review.

        From: {{ payload.sender }}
        Subject: {{ payload.subject }}
        Category: {{ analysis.category }}

        Draft preview: {{ draft.text | truncate(200) }}

        👉 Review and send: {{ saved_draft.review_url }}
```

### Schritt 2: Das Webhook-Secret einrichten

Füge das Secret zu deiner `.env`-Datei hinzu:

```bash
# .env
EMAIL_WEBHOOK_SECRET=whsec_em_4f8a2c6d9e1b3f5a7c
```

### Schritt 3: Registrieren und verbinden

```bash
openclaw automation add automations/email-responder.yaml
openclaw config reload
```

Konfiguriere jetzt deinen E-Mail-Anbieter (Mailgun, SendGrid usw.), um eingehende E-Mails weiterzuleiten an:

```
https://your-instance.openclaw.app/hooks/incoming-email
```

### Schritt 4: Mit einer simulierten E-Mail testen

```bash
openclaw automation run "email-responder" --now --payload '{
  "sender": "alice@clients.example.com",
  "subject": "Meeting next week?",
  "body": "Hi, could we schedule a call sometime next week to discuss the project? Tuesday or Wednesday would work best for me.",
  "message_id": "msg-test-001"
}'
```

```
[Step 1/4] analyze-email ............ ✓ (category: scheduling)
[Step 2/4] draft-reply .............. ✓ (247 words)
[Step 3/4] save-draft ............... ✓ (draft saved)
[Step 4/4] notify-me ................ ✓ (notification sent)
```

Der Filter-Block hält die Dinge fokussiert: Nur E-Mails von `@clients.example.com`, die *nicht* als URGENT markiert sind, lösen diese Automation aus. Alles andere wird ignoriert.

## Automation 3: Projekt-Gesundheitsmonitor

Diese Automation läuft alle vier Stunden an Geschäftstagen. Sie prüft mehrere Metriken eines Projekts — offene Issues, Build-Status, Verfügbarkeit — und sendet einen Alarm nur, wenn etwas nicht stimmt. Kein Alarm bedeutet, dass alles in Ordnung ist.

### Schritt 1: Die Automationsdatei erstellen

Erstelle `automations/project-monitor.yaml`:

```yaml
name: "project-monitor"
description: "Checks project health metrics and alerts on problems"
version: 1

trigger:
  type: "cron"
  cron: "0 8,12,16,20 * * 1-5"
  timezone: "UTC"

config:
  project_repo: "your-org/main-project"
  uptime_url: "https://api.example.com/health"
  alert_channel: "slack"

steps:
  - id: "check-issues"
    action: "github.get_issues"
    params:
      repo: "{{ config.project_repo }}"
      state: "open"
      labels: ["bug", "critical"]
    output: "critical_issues"

  - id: "check-build"
    action: "github.get_workflow_status"
    params:
      repo: "{{ config.project_repo }}"
      workflow: "ci.yml"
      branch: "main"
    output: "build_status"

  - id: "check-uptime"
    action: "http.get"
    params:
      url: "{{ config.uptime_url }}"
      timeout: 10
    output: "uptime_check"

  - id: "evaluate-health"
    action: "assistant.analyze"
    params:
      input:
        critical_issues: "{{ critical_issues }}"
        build: "{{ build_status }}"
        uptime: "{{ uptime_check }}"
      instructions: |
        Evaluate the project health based on these inputs.
        Flag as "unhealthy" if any of these are true:
        - More than 3 critical/bug issues are open
        - The latest CI build failed
        - The uptime check returned a non-200 status
        Otherwise, flag as "healthy".
    output: "health_report"

  - id: "alert-if-needed"
    action: "assistant.send_message"
    condition: "health_report.status == 'unhealthy'"
    params:
      channel: "{{ config.alert_channel }}"
      template: |
        ⚠️ Project Health Alert — {{ now | date_format }}

        Status: {{ health_report.status | upper }}

        {% if critical_issues | length > 3 %}
        🐛 {{ critical_issues | length }} critical issues open (threshold: 3)
        {% endif %}

        {% if build_status.conclusion != 'success' %}
        🔴 Latest CI build: {{ build_status.conclusion }}
           Run: {{ build_status.html_url }}
        {% endif %}

        {% if uptime_check.status_code != 200 %}
        🌐 Uptime check failed (HTTP {{ uptime_check.status_code }})
        {% endif %}

        {{ health_report.summary }}
```

### Schritt 2: Registrieren

```bash
openclaw automation add automations/project-monitor.yaml
```

### Schritt 3: Alle Verbindungen überprüfen

```bash
openclaw automation check "project-monitor"
```

```
Checking automation "project-monitor"...
  ✓ GitHub token valid (repo: your-org/main-project)
  ✓ HTTP endpoint reachable (https://api.example.com/health)
  ✓ Slack channel connected
  ✓ Cron schedule valid (next: Today 16:00 UTC)

All checks passed.
```

### Schritt 4: Einen Probelauf durchführen

```bash
openclaw automation run "project-monitor" --now --dry-run
```

Das `--dry-run`-Flag führt alle Schritte aus, sendet aber nicht tatsächlich den Alarm. Es zeigt dir, was *passiert wäre*:

```
[Step 1/5] check-issues ............. ✓ (2 critical issues)
[Step 2/5] check-build .............. ✓ (conclusion: success)
[Step 3/5] check-uptime ............. ✓ (HTTP 200)
[Step 4/5] evaluate-health .......... ✓ (status: healthy)
[Step 5/5] alert-if-needed .......... ⏭ SKIPPED (condition not met)

DRY RUN: No messages sent. Project is healthy.
```

Beachte das `condition`-Feld beim letzten Schritt. Der Alarm löst nur aus, wenn Dinge ungesund sind. Das bedeutet, du wirst nicht viermal täglich mit "alles ist in Ordnung"-Nachrichten überschwemmt.

## Was du gelernt hast

Über diese drei Automationen hinweg hast du verwendet:

- **Cron-Trigger** für zeitbasierte Planung
- **Webhook-Trigger** für ereignisgesteuerte Reaktionen
- **Mehrstufige Workflows** mit zwischen den Schritten weitergegebenen Daten
- **Bedingte Logik**, um Alarme nur bei Bedarf zu senden
- **Filter**, um einzugrenzen, welche Ereignisse die Automation auslösen
- **Vorlagen** für lesbare, dynamische Ausgabe
- **Probeläufe** und Testbefehle für sichere Entwicklung

Jede Automation in OpenClaw folgt derselben Struktur: Trigger, optionaler Filter, Schritte, optionale Bedingungen. Sobald du mit diesen Bausteinen vertraut bist, kannst du sie kombinieren, um praktisch jeden Workflow zu bewältigen, den du dir vorstellen kannst.

## Nächste Schritte

- Stöbere in der [Cron-Jobs](/docs/automation/cron-jobs)-Referenz für weitere Planungsmuster.
